## 二、核心架构差异

### OpenClaw 架构（Gateway 中心化）

```
用户消息（WhatsApp/Telegram/Slack/Discord/...）
               │
               ▼
┌───────────────────────────────┐
│       Gateway（控制平面）        │
│     ws://127.0.0.1:18789      │
└──────────────┬────────────────┘
               │
    ┌──────────┼──────────┐
    │          │          │
  Pi Agent   Browser    Nodes
  (AI推理)   (CDP控制)  (iOS/Android/macOS)
```

**关键点：** Gateway 是单一控制平面，所有客户端（CLI/App/Web）通过 WebSocket 连接。
浏览器控制通过 **本地 HTTP API + Playwright** 实现，不依赖外部服务。

### TestPilot AI 架构（VS Code 插件 + Python 引擎）

```
VS Code 插件 (TypeScript)
       │
       ▼
Python 引擎 (FastAPI)
       │
    ┌──┼──────┬──────────┬────────────┐
    │  │      │          │            │
  Web  │   Android    iOS       Desktop
  Playwright  Appium+U2  Appium+XC  PowerShell
```

**关键点：** 插件负责蓝本管理和AI规则注入，Python引擎负责执行测试。
每个平台有独立的 Runner 实现。

## 三、浏览器点击精准度对比（核心差异）

### 3.1 OpenClaw 的 Snapshot → Ref → Locator 三步法

OpenClaw 浏览器控制的核心是 **Snapshot Ref 系统**，这是它点击精准的根本原因：

```
第1步：Snapshot（获取页面结构快照）
  ├─ AI Snapshot（默认）：Playwright 内置的 aria-ref 标注系统
  │   → 每个可交互元素自动编号：ref=1, ref=2, ref=3 ...
  │   → 返回文本化的 UI 树，附带语义信息
  └─ Role Snapshot（高级）：基于 ARIA role 的结构化快照
      → 每个元素编号为 e1, e2, e3 ...
      → 通过 getByRole() 精确定位

第2步：AI 选择 Ref 编号
  → AI 看到快照文本，直接说 "click ref=12"
  → 不需要猜测坐标，不需要理解 CSS

第3步：refLocator() 解析 Ref → Playwright Locator
  → ref=12 → page.locator('aria-ref=12') → Playwright 精确点击
  → ref=e5 → getByRole('button', { name: '提交', exact: true }) → 精确点击
```

**源码关键函数 `refLocator()`（pw-session.ts）：**
```typescript
export function refLocator(page: Page, ref: string) {
  // e12 格式 → Role-based 定位
  if (/^e\d+$/.test(normalized)) {
    // aria 模式：直接用 aria-ref
    if (state?.roleRefsMode === "aria") {
      return scope.locator(`aria-ref=${normalized}`);
    }
    // role 模式：通过 getByRole 精确匹配
    const info = state?.roleRefs?.[normalized];
    const locator = info.name
      ? locAny.getByRole(info.role, { name: info.name, exact: true })
      : locAny.getByRole(info.role);
    return info.nth !== undefined ? locator.nth(info.nth) : locator;
  }
  // 数字格式 → AI Snapshot 的 aria-ref
  return page.locator(`aria-ref=${normalized}`);
}
```

**OpenClaw 点击流程（clickViaPlaywright）：**
```typescript
// 1. AI 给出 ref 编号
// 2. refLocator 解析为 Playwright Locator
const locator = refLocator(page, requireRef(resolved.ref));
// 3. Playwright 内部精确定位 DOM 元素并点击
await locator.click({ timeout, button, modifiers });
```

### 3.2 TestPilot AI 的 CSS 选择器直连法

TestPilot Web 端的点击流程要简单得多：

```python
# blueprint_runner.py → _execute_step()
# 1. 蓝本中写死 CSS 选择器：#login-btn, .submit-btn
# 2. 直接传给 Playwright
await self._browser.click(target)  # target = "#login-btn"

# automator.py → click()
await self.page.click(selector)  # 直接 CSS 选择器
```

### 3.3 精准度差异根因

| 维度 | OpenClaw | TestPilot |
|------|----------|-----------|
| **定位方式** | 运行时 Snapshot → Ref → Locator | 编译时静态 CSS 选择器 |
| **选择器来源** | AI 实时从页面快照中选择 | 编程AI从源码中提取 |
| **容错性** | Ref 每次刷新，始终对应当前页面元素 | CSS 选择器可能因代码变动过时 |
| **动态内容** | AI 看到实际渲染的页面结构 | 基于源码静态分析，可能遗漏动态DOM |
| **元素可见性** | Playwright Locator 自动等待可见 | 同样使用 Playwright，但选择器可能不精确 |

## 四、移动端控制对比

### OpenClaw 的移动端方案

OpenClaw **不做移动端UI自动化测试**。它的 Android/iOS 节点是 **伴侣App**：

- Android 节点：原生 Kotlin App，通过 WebSocket 连接 Gateway
- 只提供 **设备命令**：camera.snap、notifications.list、contacts.search、sms.search 等
- **Canvas 展示**：Android WebView 加载 Gateway 上的 HTML Canvas
- **不操作第三方App**：不能打开淘宝、不能点击微信消息

```
OpenClaw Android 能做的：
  ✅ 拍照 (camera.snap)
  ✅ 读通知 (notifications.list)
  ✅ 搜联系人 (contacts.search)
  ✅ 语音对话 (Voice Tab)
  ✅ 在Canvas中显示AI生成内容

OpenClaw Android 不能做的：
  ❌ 操作任意第三方App界面
  ❌ 精确点击App中的按钮
  ❌ 在App表单中填写内容
  ❌ 自动化UI测试
```

### TestPilot 的移动端方案

TestPilot 的移动端使用 **Appium + UIAutomator2** 做真正的UI自动化：

```
双保险策略：
  第1层：UI树匹配（Appium /source → XML → bounds 中心坐标）
    - accessibility_id:xxx → content-desc 匹配
    - //android.widget.EditText[@hint='xxx'] → XPath 属性匹配
    - id:xxx → resource-id 匹配

  第2层：AI 截图分析（截图 → 发给AI → 归一化坐标 → 屏幕坐标）
    - 发送截图 + 目标元素描述给 AI
    - AI 返回归一化坐标 (0~1)
    - 转换为设备像素坐标并 tap
```

**TestPilot 移动端的问题：**

| 问题 | 原因 | 影响 |
|------|------|------|
| AI 坐标不精准 | AI 视觉分析有误差，尤其是密集UI | 点错元素或点不到 |
| UI树经常匹配不到 | Flutter/自绘应用无标准 Accessibility 属性 | 被迫降级到 AI 视觉 |
| Session 重建耗时 | U2 崩溃后重建 Session 需 20-40 秒 | 测试速度慢 |
| 页面指纹依赖文本 | Jaccard 相似度 ≥ 60% 才匹配 | 相似页面可能误匹配 |

### 关键结论

**OpenClaw 在移动端不做 UI 自动化，所以不存在"点得准"的问题。**
TestPilot 做的是更难的事（操作任意 App 界面），自然面临更多挑战。

## 五、桌面端控制对比

### OpenClaw 桌面端

- macOS 原生菜单栏 App + `system.run`（执行 shell 命令）
- `system.notify` 发通知
- **不操作桌面应用的UI元素**（不点击按钮、不填表单）
- Canvas 做可视化展示（A2UI 推送 HTML/CSS/JS）

### TestPilot 桌面端

- 双保险策略（与移动端类似）：
  1. UI Automation（PowerShell + .NET）精确查找控件
  2. AI 截图分析降级（截图 hash 缓存 + 归一化坐标）
- **真正操作桌面应用**：点按钮、填输入框、断言文本

**TestPilot 桌面端的问题同移动端：** AI 坐标有误差，tkinter/Electron 等框架的
UI Automation 属性很差，经常被迫降级到 AI 视觉分析。

## 六、OpenClaw 可借鉴的技术方案

### 6.1 🔥 Snapshot Ref 系统（Web 平台最值得借鉴）

**借鉴方向：** TestPilot Web 端可引入 Snapshot → Ref → Locator 机制作为
CSS 选择器的 **容错降级方案**。

**具体实施：**
```
当前流程（不变）：
  蓝本 CSS 选择器 → Playwright click(selector) → 成功

新增降级机制：
  蓝本 CSS 选择器 → Playwright click(selector) → 失败！
    ↓
  触发 Snapshot Ref 降级：
    1. 调用 Playwright page.accessibility.snapshot() 获取可访问性树
    2. 提取所有可交互元素及其 role/name
    3. 匹配蓝本中 step.description 描述的目标元素
    4. 用 getByRole(role, { name }) 精确定位
    5. 重试操作
```

**投入产出比：** ★★★★★ — Playwright 已内置 ARIA Snapshot 能力，
只需添加约 50 行降级逻辑，无需额外依赖。

### 6.2 🔥 Labels 截图标注（移动/桌面端可借鉴）

OpenClaw 的 `screenshotWithLabelsViaPlaywright()` 会在截图上叠加元素编号标签：

```typescript
// 在页面 DOM 上注入标注层
const box = document.createElement("div");
box.style.border = "2px solid #ffb020";  // 黄色边框标出元素
const tag = document.createElement("div");
tag.textContent = label.ref;  // 显示编号 e1, e2...
tag.style.background = "#ffb020";
```

**借鉴方向：** TestPilot 移动/桌面端 AI 分析截图前，先在截图上用 OpenCV
标注 UI 树中识别到的元素边框和编号，再发给 AI。
这样 AI 能更精确地定位元素，而不是在无标注的截图上猜测坐标。

**投入产出比：** ★★★★☆ — 需要 OpenCV 绘图，约 80 行代码。

### 6.3 结构化 DOM 上下文（Web 端已部分实现）

OpenClaw 的 AI Snapshot 返回结构化的 UI 树文本，AI 从中选择元素。
TestPilot Web 端的 L2 诊断已经有类似机制（`dom_context`）：

```python
# blueprint_runner.py L280+ — 已有的 DOM 上下文收集
dom_ctx = await self._browser.page.evaluate("""() => {
    const tags = ['button','input','select','textarea','a'];
    const els = tags.flatMap(tag => [...document.querySelectorAll(tag)].map(el => {
        const text = (el.innerText || el.value || el.placeholder || '').trim();
        return {tag, text, id: el.id || '', cls: el.className};
    })).filter(Boolean).slice(0, 30);
    return JSON.stringify(els);
}""")
```

**借鉴方向：** 将此 DOM 上下文从 L2 诊断阶段提前到 **首次失败即刻使用**，
结合 ARIA 快照做智能选择器修复，而不是等 AI 中枢多轮决策。

**投入产出比：** ★★★☆☆ — 逻辑已有基础，需要重构调用时机。

### 6.4 Batch Actions（性能优化）

OpenClaw 支持 `batch` 动作，一次请求执行多步操作：
```typescript
case "batch":
  await batchViaPlaywright({
    actions: action.actions,  // 多个 click/type/wait 打包执行
    stopOnError: action.stopOnError,
  });
```

**借鉴方向：** TestPilot 当前逐步执行，每步都有截图/验证开销。
对于连续的 fill 操作（如填写表单多个字段），可以合并为批量操作，减少通信延迟。

**投入产出比：** ★★☆☆☆ — 优化不大，测试场景需要逐步验证。

### 6.5 Evaluate 安全控制

OpenClaw 的 `evaluate` 有完善的安全机制：
- 可通过 `browser.evaluateEnabled=false` 全局禁用
- 超时控制（`Promise.race` + `evaluateTimeout`）
- AbortSignal 支持强制断开 CDP 连接

**借鉴方向：** TestPilot 的 `page.evaluate()` 调用缺少超时保护，
如果页面 JS 挂起会永久阻塞。应添加超时和取消机制。

**投入产出比：** ★★★☆☆ — 安全加固，约 20 行代码。

## 八、行动优先级

1. **[P0] Web 端 ARIA Snapshot 降级** — CSS 选择器失败时自动用 Playwright
   accessibility.snapshot() + getByRole() 重试，成本最低收益最大
2. **[P1] AI 截图标注** — 移动/桌面端截图发 AI 前，用 UI 树 bounds
   在图上画框+编号，提升 AI 坐标分析精度
3. **[P2] DOM 上下文前置** — Web 端首次失败即收集 DOM 上下文辅助修复，
   不等 L2 决策
4. **[P3] JS evaluate 超时保护** — 添加 Promise.race 超时机制，防止页面
   挂起阻塞测试

## 九、核心认知纠偏

> **OpenClaw ≠ RPA 自动化工具。** 它是一个 **AI 聊天助手**，浏览器控制
> 只是它的一个工具（tool）。它不能操作任意手机 App、不能在桌面应用中
> 点击按钮、不能测试小程序。
>
> **TestPilot 做的事情比 OpenClaw 难得多。** TestPilot 要在 5 个完全不同
> 的平台上精确操作 UI 元素，而 OpenClaw 只需要控制一个受管理的 Chrome 实例。
>
> **但 OpenClaw 在 Web 浏览器控制上确实更精准。** 核心原因是 Snapshot Ref
> 系统——运行时感知页面结构，而不是依赖静态源码分析生成的 CSS 选择器。
> 这一点值得 TestPilot Web 端借鉴。

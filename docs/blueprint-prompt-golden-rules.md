## 测试设计黄金8规则（必须严格遵守）

### 规则1：功能全覆盖

- **先通读全部源代码**，列出所有功能点（每个按钮、每个表单、每个Tab、每个弹窗、每个下拉框）
- 每个功能点必须至少有一个测试场景，不能遗漏任何可操作的UI元素
- **自检清单**：数一数代码里有多少个按钮/表单/页面，蓝本里是否每个都覆盖到了
- 如果有10个按钮，蓝本里就至少要有10个click + 10个断言

### 规则2：操作→断言配对（最核心的规则）

- **每一个操作**（click / fill / select）**后面必须跟一个断言**（assert_text / assert_visible / screenshot）验证结果
- 没有断言的操作 = 没测试

| ❌ 错误示范 | ✅ 正确示范 |
|------------|-----------|
| click 登录按钮 → 结束 | click 登录按钮 → assert_text 验证"欢迎回来" |
| fill 搜索框 → 结束 | fill 搜索框 → click搜索 → assert_text 验证搜索结果 |
| click 删除按钮 → 结束 | click 删除按钮 → assert_text 验证"删除成功" → 验证列表少了一条 |

### 规则3：业务流程端到端串联

- 除了单点功能测试，**必须有完整业务流程场景**
- 每个流程场景至少串联3个以上页面/功能
- 示例：

```
场景：完整购物流程
  注册账号 → 登录 → 浏览商品列表 → 搜索商品 → 加入购物车
  → 查看购物车 → 修改数量 → 填写收货地址 → 选择配送方式
  → 使用优惠券 → 提交订单 → 查看订单详情
```

### 规则4：状态变化验证

- 操作前先读取当前状态值，操作后再读取，对比变化是否符合预期

| 场景 | 操作前 | 操作 | 操作后断言 |
|------|--------|------|-----------|
| 加入购物车 | 读取购物车数量=0 | click加入购物车 | 断言购物车数量=1 |
| 删除商品 | 读取列表3条 | click删除 | 断言列表2条 |
| 修改数量 | 读取总价=100 | fill数量=2 | 断言总价=200 |

### 规则5：异常和边界测试

- **每个表单必须测试**：
  - 空提交（什么都不填直接点提交，验证错误提示）
  - 超长输入（200+字符，验证是否截断或报错）
  - 特殊字符（`<script>alert(1)</script>`、`' OR 1=1`）
  - 格式错误（邮箱格式不对、手机号太短）
- **每个需要权限的操作**：
  - 未登录直接访问，验证跳转到登录页或提示"请先登录"
  - 无权限操作，验证提示"权限不足"

### 规则6：弹窗和提示验证

- 操作后出现的**成功提示、错误提示、确认弹窗**，必须用断言验证内容
- 不只是验证弹窗出现了，更要验证**弹窗里的文字是否正确**

| 场景 | 要验证的内容 |
|------|-------------|
| 提交订单成功 | 验证提示文字包含"下单成功"或"订单已提交" |
| 删除操作 | 验证确认弹窗文字包含"确定删除" |
| 表单校验失败 | 验证错误提示文字（如"请输入用户名"） |
| 登录失败 | 验证错误提示（如"密码错误"） |

### 规则7：选择器规范

- **使用代码中的真实 id 或稳定 class**
- **禁止使用脆弱选择器**

| ❌ 禁止 | ✅ 正确 |
|--------|--------|
| `div:nth-child(3)` | `#login-btn` |
| `body > div > div > span` | `.product-card` |
| `[class*="css-"]` (CSS Modules哈希) | `[data-testid="submit"]` |

- **必须先阅读源代码确认选择器存在**，不能猜测

### 规则8：启动命令

- 如果应用需要命令行启动，必须填写 `start_command` 字段

| 应用类型 | start_command 示例 |
|---------|-------------------|
| Node.js | `npm start` 或 `node server.js` |
| Python | `python app.py` 或 `uvicorn main:app` |
| Docker | `docker-compose up` |
| 纯HTML静态 | 留空（引擎内置预览服务器） |

## 蓝本JSON基本结构

```json
{
  "app_name": "应用/模块名称",
  "description": "50-200字功能说明，描述本蓝本覆盖哪些功能和测试范围",
  "base_url": "http://localhost:端口",
  "version": "1.0",
  "platform": "web | miniprogram | android | desktop",
  "start_command": "启动命令（纯HTML留空）",
  "start_cwd": "./",
  "pages": [
    {
      "url": "/页面路径",
      "title": "页面标题",
      "elements": {
        "元素描述": "#实际CSS选择器"
      },
      "scenarios": [
        {
          "name": "场景名称",
          "description": "场景说明",
          "steps": [
            {"action": "动作类型", "target": "选择器", "value": "值", "expected": "预期", "description": "步骤说明"}
          ]
        }
      ]
    }
  ]
}
```

## 规则9：深度Bug挖掘（安全审计视角）

请以**资深QA架构师 + 安全审计师**的双重视角审视代码，主动挖掘以下隐蔽Bug：

| 审计维度 | 检查要点 | 示例场景 |
|----------|----------|----------|
| 认证绕过 | 密码是否真正校验？是否存在硬编码/万能密码？ | 用随机密码尝试登录，断言应失败 |
| 数值精度 | 价格计算是否用浮点加法？是否做了精度处理？ | 12.8+6.5 应等于 19.30 而非 19.299999 |
| 库存/数量边界 | 加购是否校验库存上限？数量能否为负/0/超大？ | 库存50的商品加购51次，断言被拦截 |
| 优惠/折扣逻辑 | 优惠金额是否真正从总价中扣除？ | 首单减5元：总价应为原价-5，断言实付金额 |
| 配送/运费逻辑 | 不同配送方式运费是否正确？免费条件是否生效？ | 选"到店自提"后，断言运费=0且总价不含运费 |
| 状态一致性 | 前端显示文字与实际计算值是否一致？ | 显示"免费"但实际算了3元 |
| 并发/重复提交 | 按钮有无防重复点击？表单能否重复提交？ | 快速双击提交按钮，断言不会创建两个订单 |

**关键**：对于每个可疑逻辑，设计专门的验证场景——先构造触发条件，再用 `assert_text` 断言实际计算结果。

## 规则10：参考功能文档

- 如果项目目录下存在 **README.md、需求文档、功能说明、CHANGELOG** 等文件，必须先阅读
- 根据文档中描述的功能清单逐一核对：文档说有的功能，蓝本必须覆盖
- 根据文档中的业务规则设计断言（如"满100免运费" → 构造99元和100元两个场景分别验证）
- 如果没有文档，则从**代码注释、变量命名、函数名**中推断业务意图，对比实际行为是否一致

## 规则11：截图策略（省钱省时）

截图用于发现**视觉Bug**（布局溢出、元素遮挡、样式错乱、响应式崩坏、图片加载失败），断言无法检测这类问题。

### 截图规则

- **每个蓝本模块的第1个场景末尾加1张 `screenshot`**，其余场景不加
  - `auth.testpilot.json` 第1个场景末尾 → `screenshot`（覆盖登录页视觉）
  - `cart.testpilot.json` 第1个场景末尾 → `screenshot`（覆盖购物车视觉）
  - 同模块的第2、3、4…个场景 → 不加
- 断言失败时引擎会自动截图留证，蓝本不用额外写
- ❌ **禁止**每个场景末尾都加 `screenshot`（浪费视觉大模型费用和时间）

## 规则12：蓝本按功能模块拆分

- 按**业务功能模块**拆分蓝本，不要把所有场景塞进一个文件
- 拆分原则：**一个功能模块 = 一个蓝本文件**，改了哪个功能就只跑哪个蓝本
  - 例：`auth`（登录注册）、`cart`（购物车）、`checkout`（结算支付）、`profile`（个人中心）
  - 跨页面的端到端流程归属到终点功能模块（如"加购→结算→下单"归入 `checkout`）
- 命名规范：`testpilot/<模块名>.testpilot.json`
- 每个蓝本独立可运行，场景开头都要从 `navigate` 开始
- 全量测试时通过 `run_blueprint_batch` 批量运行所有蓝本

## 规则13：蓝本增量修改（省Token）

- 已有蓝本时，只修改/新增变更涉及的场景，不要重写整个文件
- 新增功能 → 新增场景追加到对应模块蓝本
- 修改功能 → 更新对应场景的步骤和断言
- 删除功能 → 删除对应场景
- **禁止**每次都从零生成全新蓝本覆盖旧文件

## 自检清单（生成蓝本后必须逐项核对）

- [ ] 是否通读了全部源代码？
- [ ] 代码里有多少个按钮？蓝本里是否每个都click + assert了？
- [ ] 代码里有多少个表单？每个字段是否都fill + assert了？
- [ ] 代码里有多少个页面/Tab？是否每个都navigate + assert了？
- [ ] 是否有至少一个端到端完整业务流程场景（≥3个页面串联）？
- [ ] 是否有异常场景（空提交、错误输入、未登录）？
- [ ] 每个操作后面是否都跟了断言？（没有断言的操作 = 没测）
- [ ] 所有选择器是否来自源代码的真实id/class？
- [ ] platform字段是否正确？
- [ ] start_command是否正确（需要启动的应用必填）？
- [ ] 是否以安全审计视角检查了认证/数值精度/库存边界/优惠逻辑？
- [ ] 是否阅读了项目文档（README/需求/CHANGELOG）并据此设计了断言？
- [ ] 截图是否只用于视觉Bug场景和关键结果页？（禁止每个场景末尾都截图）
- [ ] 蓝本是否按功能模块拆分？（禁止单个文件几百步）
- [ ] 是否在原有蓝本基础上增量修改？（禁止每次从零重写）

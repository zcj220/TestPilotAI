## 一、项目愿景

打造一个AI驱动的自动化测试机器人，能够：
- 像人类一样操作应用界面（点击、输入、滑动）
- 自动发现Bug并生成详细报告
- 自动修复发现的问题并重新测试
- 让不懂技术/不懂特定领域的用户也能验证AI生成的代码

**核心价值**：补全AI编程工具链的"最后一公里"——验证。

## 二、已确认的核心设计要点

### 2.1 三AI交叉验证架构（创新点）

- **AI-A（编程AI）**：写出应用
- **AI-B（数据AI）**：独立生成测试数据（不同对话窗口，避免自我一致性偏差）
- **AI-C（测试AI）**：在沙箱中模拟人类操作，用B的数据测试A的应用
- 三个AI互相独立，不共享上下文，避免"自己考自己"

### 2.2 Docker沙箱环境（"桶"）

- 使用Docker Sandbox或AIO Sandbox作为隔离执行环境
- AI在沙箱内有完全权限，随意操作不影响宿主机系统
- 沙箱内集成：浏览器、Shell、文件系统、开发工具

**已确定方案：编程在本地，测试在桶里（方案A）**
- 用户照常用Cursor/Windsurf在本地编程
- 通过Docker Volume Mount将项目文件夹挂载到沙箱（如 `-v D:\Projects\MyApp:/workspace`）
- 文件实时同步，无需打包/部署步骤，AI全自动完成
- 沙箱内自动执行启动命令（如 `npm run dev`）运行应用
- 用户仅需首次配置：项目路径 + 启动命令

**关键约束**：
- 必须兼容市面上各种AI编程工具（Cursor、Copilot、Windsurf、TRAE等）
- 编程工具在本地运行，完全不受沙箱影响，零侵入

### 2.3 测试流程：用户审批制

1. 用户输入完整需求描述
2. AI生成测试脚本（含每步的预期结果）
3. 用户审批测试脚本（确认步骤和预期结果是否正确）
4. AI按脚本自动执行，逐步对比实际结果 vs 预期结果
5. 不一致的步骤记录为Bug
6. 生成完整测试报告，用户查看

**测试脚本格式示例**：
```
步骤1: 打开 /login 页面
  -> 预期: 显示登录表单，含用户名、密码、登录按钮

步骤2: 输入用户名"admin"，密码"123456"，点击登录
  -> 预期: 跳转到 /dashboard，显示欢迎信息
```

### 2.4 多模态分析：豆包视觉API

- 测试执行时录屏 + 关键帧提取（UI状态变化时截帧）
- 截帧发送给豆包视觉理解API进行分析
- 价格：0.003元/千token，1元可处理284张720P图片
- 每次完整30分钟测试的多模态分析成本约 0.75-2元

### 2.5 自动修复闭环（核心价值）

- 测试发现Bug -> 生成结构化Bug报告（含截图、实际vs预期、错误上下文）
- 交给编程AI自动修复 -> 修复后重新执行测试
- 通过 -> 标记"已自动修复，请确认"
- 未通过或无法修复 -> 标记"需人工介入"

### 2.6 记忆系统（SQLite本地存储）

- ~~原计划用Mem0，实际采用更轻量的SQLite方案~~
- 存储：测试历史、经验教训、页面指纹（`data/memory.db`）
- 零外部依赖，无需额外LLM后端
- 后续计划：历史记录定期压缩为经验总结

## 六、已决策事项（第三轮讨论）

### 6.1 多模态分析策略
- **每步截帧 + 事后视频录像作为补充**
- 每执行一个测试步骤截一帧，用于和预期结果精确对比
- 整个测试过程同时录屏，视频留存作为证据和回溯用
- 具体参数待开发中根据API调用成本动态调整

### 6.2 测试数据领域覆盖
- 当前无成熟的"领域测试数据生成API"可直接接入
- **第一阶段**：先用大模型自身知识生成测试数据
- **后续优化**：针对常见领域（财务、医疗、电商）准备提示词模板库
- **远期**：支持用户上传领域文档（如公司内部业务规则），AI参考生成更精准测试数据

### 6.3 测试遇Bug时的行为：分类处理
- **阻塞性Bug**（页面崩溃、无法进入下一步）→ 自动停下，等待修复后重测
- **非阻塞性Bug**（文字错误、布局偏移）→ 记录后继续执行
- 所有分类Bug跑完或跑不下去后，统一交给编程AI修复

### 6.4 技术选型（已确定）

**推荐技术栈**：

| 层 | 技术 | 选择理由 |
|----|------|---------|
| 核心引擎 | **Python + FastAPI** | AI生态最好，LangChain/Mem0/OpenAI SDK均Python优先 |
| 浏览器自动化 | **Playwright (Python)** | 支持Chromium/Firefox/WebKit，原生截图录屏，跨平台 |
| Docker管理 | **docker-py** | Python原生Docker SDK，成熟稳定 |
| 记忆系统 | **SQLite**（原计划Mem0） | 轻量、零依赖、本地存储 |
| IDE插件 | **TypeScript + VSCode Extension API** | Windsurf/VSCode通用，同一份代码 |
| 独立桌面应用 | **Tauri (Rust壳) + React + TypeScript** | 安装包3-10MB（Electron需50MB+），内存低，启动快 |
| 前端UI | **React + TailwindCSS** | IDE插件和桌面应用共享UI组件库 |
| 通信协议 | **HTTP/WebSocket (FastAPI ↔ 前端)** | 跨语言通信，前后端解耦 |

**Tauri vs Electron选型依据**：
| 维度 | Tauri | Electron |
|------|-------|----------|
| 安装包大小 | 3-10MB | 50MB+ |
| 内存占用 | 低（系统WebView） | 高（捆绑Chromium） |
| 启动速度 | 快 | 慢 |
| 安全性 | 高（Rust内存安全） | 中 |
| 跨平台一致性 | 中（WebView有差异） | 高 |
| 生态成熟度 | 中（增长快） | 高（老牌） |
| 更新维护 | 轻量，更新快 | 更新包大 |

选Tauri因为：桌面应用只是展示层（报告、配置、日志），重逻辑在Python引擎，需要应用尽量轻量，用户电脑同时还跑IDE+Docker+AI工具。

**系统架构**：
```
前端展示层
  ├── IDE插件: TypeScript + VSCode Extension API
  └── 桌面应用: Tauri (Rust) + React + TypeScript
      └── 共享UI组件库: React + TailwindCSS
              │
              │ HTTP / WebSocket
              ▼
核心引擎层: Python
  ├── FastAPI (HTTP服务)
  ├── AI调度: LangChain / 豆包API直调
  ├── 记忆系统: Mem0
  ├── Docker管理: docker-py
  └── 浏览器自动化: Playwright (Python)
```

**支持平台**：Windows、macOS、Linux（三平台，Tauri和Playwright均原生支持）

### 7.5 依赖管理原则（重要）
1. **锁定所有版本号** — 用 poetry.lock / package-lock.json 锁死版本
2. **使用虚拟环境** — Python用poetry或venv隔离
3. **最小化依赖** — 能不引入的库不引入
4. **提前验证兼容性** — 每加一个新依赖先在干净环境验证
5. **开发前先搭建并验证完整依赖链** — 确保所有库能一起安装运行

### 7.6 记忆系统：SQLite本地存储（v1.3已完整激活）
- ~~原计划用Mem0，但实际开发中发现需求较简单，用SQLite即可~~
- 当前实现：`src/memory/store.py`（5张表）+ `src/memory/compressor.py`（规则压缩器）
- 存储文件：`data/memory.db`（自动创建）
- 5张表：test_history、experiences、page_fingerprints、bug_patterns（v1.3）、fix_experiences（v1.3）
- 记忆提取：MemoryCompressor从BugReport/FixPlan固定字段直接提取，零AI成本
- 记忆检索：get_context_for_url()分层查询（Bug模式+修复经验+通过率趋势），≤500字符注入提示词
- 淘汰机制：cleanup()清理30天过期记录 + 同URL最多10条Bug模式
- 设计理念：格式化是前提→字段直接提取→零AI成本→经验=Bug发现+修复方案

## 八、已实现：智能修复模式（v1.3）

### 单一智能模式（取代原双模式设计）

经评估，"逐个修复"和"批量修复"各有优劣，最终采用**智能模式**——由 `SmartRepairDecider` 自动判断每个Bug的修复策略，用户无需选择：

**阻塞性Bug → 立即修复（IMMEDIATE）**
判断标准（满足任一即触发）：
1. Bug严重度为 CRITICAL
2. 步骤执行状态为 ERROR（浏览器操作失败）
3. 连续失败步骤数 ≥ 3
4. Bug涉及白屏/崩溃关键词

**非阻塞性Bug → 延迟批量修复（DEFERRED）**
- 计算错误、样式问题、文案错误等
- 攒到测试全部执行完后，一次性批量修复
- 自然去重（同类Bug不重复修）

**实现文件**
- `src/testing/smart_repair.py` — SmartRepairDecider 决策器
- `src/testing/blueprint_runner.py` — run() 方法集成 repair_callback
- 报告末尾自动附带智能修复统计

## 九、已决策：两种测试模式（核心架构）

### 模式一：蓝本模式（精确功能测试）—— 优先实现

**核心思路：编程AI生成代码时，必须同时输出测试蓝本**

蓝本 = 测试说明书，由编程AI（Windsurf/Cursor等）生成，内容包括：
- 页面有哪些元素、CSS选择器是什么
- 每个按钮/输入框的功能和预期效果
- 测试场景和步骤（点哪里、输什么、预期看到什么）

**蓝本格式示例（testpilot.json）：**
```json
{
  "app_name": "Todo待办应用",
  "pages": [
    {
      "url": "/",
      "title": "Todo 待办事项",
      "elements": {
        "输入框": "#todoInput",
        "添加按钮": "#addBtn",
        "待办列表": "#todoList"
      },
      "scenarios": [
        {
          "name": "添加待办",
          "steps": [
            {"action": "fill", "target": "#todoInput", "value": "测试任务"},
            {"action": "click", "target": "#addBtn"},
            {"expected": "列表中出现'测试任务'"}
          ]
        }
      ]
    }
  ]
}
```

**工作流程：**
1. 用户让编程AI写完功能后，要求它输出 `testpilot.json` 蓝本
2. TestPilot AI 读取蓝本，按精确选择器执行测试
3. 100%不会猜错选择器，专注于功能验证
4. 适用场景：开发中、提交前、功能验证

**为什么蓝本模式优先：**
- 当前"盲测"模式AI猜选择器经常猜错（实测验证，`data-testid` vs `#todoInput`）
- 用户最高频的需求是"确认功能是否正确"，蓝本模式最精准
- 编程AI完全有能力输出蓝本，只需要一段提示词引导

### 模式二：意外模式（AI自由探索测试）

**核心思路：AI自由操作页面，发现设计之外的Bug**

**方案A（优先实现）：批量截图分析**
- AI自由操作页面，连续执行10步
- 每步只截图+记录操作日志，不调AI分析（快、零成本）
- 10步结束后，10张截图+操作日志打包，一次性发AI分析
- 发现异常再做深度分析
- 一轮10步只调AI 1-2次，速度快、成本低

**方案B（备选）：视频分析模式**
- 录制几分钟操作视频
- 发给 Doubao-Seed-1.8 视频理解能力分析
- 模型支持低帧率超长视频理解
- 成本：输入0.8元/百万tokens，预估1分钟视频几分钱到一毛钱
- 方案A不够完善时补充使用

**使用场景：**
- 蓝本测试通过后，夜间/闲时放着跑
- 不需要用户盯着看，后台自动执行
- 发现的意外Bug汇总到报告中

1. **v1.x**：蓝本模式（解析 testpilot.json + 精确执行）

## 十、已决策：蓝本详细规范与异常上报机制

### 蓝本应该写多详细？

蓝本不只是"点哪里"，而是完整的**操作→等待→验证**三元组：

**每个步骤必须包含：**
1. **操作**：点击/输入/选择等
2. **等待**：操作后等多久再验证（`wait_after_ms`），处理异步加载/动画
3. **预期**：操作后应该看到什么变化

**蓝本增强字段（v1.x迭代）：**
```json
{
  "action": "click",
  "target": "#addBtn",
  "description": "点击添加按钮",
  "wait_after_ms": 500,
  "expected": "列表新增一行，显示刚才输入的文字；统计栏数字+1",
  "unexpected_is_bug": true
}
```

**智能输入：**
```json
{
  "action": "fill",
  "target": "#priceInput",
  "value": "auto:number:1-999",
  "description": "AI随机生成一个合理价格数字输入"
}
```

`auto:` 前缀表示让AI自主生成合理值：
- `auto:number:1-999` — 随机数字
- `auto:text:人名` — 随机生成一个合理的中文人名
- `auto:email` — 随机生成邮箱
- 固定值直接写字符串

**计算验证：**
```json
{
  "action": "assert_text",
  "target": "#totalPrice",
  "value": "=数量×单价",
  "description": "验证总价计算公式是否正确"
}
```

### 蓝本外异常的上报机制（通用异常检测）

**核心原则：蓝本没提到的异常也是异常，照样上报。**

编程AI写蓝本时可能有疏漏，不可能覆盖所有情况。
因此TestPilot在每步截图后，除了验证蓝本的`expected`，还自动做**通用异常检测**：

1. **页面崩溃/白屏** — 页面内容为空或出现错误页
2. **红色报错信息** — 页面上出现明显的错误提示
3. **控制台JS报错** — 浏览器控制台的error级别日志
4. **布局严重异常** — 元素重叠、溢出、消失等
5. **网络请求失败** — 关键API返回4xx/5xx

这些异常即使蓝本没写，也会作为**"蓝本外异常"**上报，标注为：
- 类型：`unexpected` 
- 说明：TestPilot自动检测，非蓝本预期范围
- 建议：请编程AI检查此异常是正常行为还是Bug

**用户/编程AI收到上报后自行判断**：
- 如果是Bug → 修复
- 如果是正常行为 → 下次在蓝本里标注"此为正常"（加入白名单）

#### v5.0 技术决策记录（2026-03-05讨论）

**问题0：手机蓝本格式和日志收集与 Web 测试是否一致？**
- **蓝本格式**：
  - 手机浏览器测试网页 → **完全相同**，CSS 选择器（`#addBtn`、`.btn`）在手机浏览器里同样精确，同一份 `testpilot.json` 直接复用，屏幕尺寸不影响选择器
  - 原生 App（APK）→ 需加 `"platform": "android"` 字段，目标改用 `resource-id`（`com.app:id/btn`）或 `accessibility-id`，属未来扩展
- **日志收集已实现 v5.0-E（已完成）**： `adb logcat` 守护线程 + Appium `get_log('browser')` + WebView 上下文自动切换，全部上线

**问题1：熄屏/锁屏时能否测试？**
- 结论：不能。UiAutomator2依赖屏幕UI元素树，熄屏后截图纯黑、操作失效。
- 方案：**自动保持亮屏（已实现）**。launch()时通过adb设置`stay_on_while_plugged_in=7`（充电时永不灭屏）+ `screen_off_timeout=2147483647`（超时最大值）+ `KEYCODE_WAKEUP`唤醒屏幕。close()时自动恢复原始值。用户零操作。

**问题2：权限弹窗如何处理（无人值守）？**
- 标准权限：Appium `autoGrantPermissions: true`（已实现），自动点击"允许"。
- adb预授权：`adb shell pm grant <pkg> <permission>`，可批量授权，无需人在场。
- 厂商定制弹窗（华为/小米/OPPO/vivo）：标准Appium无法覆盖，需维护厂商弹窗选择器库（v5.1计划）。
- 蓝本扩展：计划在蓝本中新增 `permissions` 字段，launch时自动adb授权。

**问题3：模拟器 vs 真机 vs 容器**
| 场景 | 权限 | 熄屏 | 适用 |
|------|------|------|------|
| 容器内模拟器 | ✅ 完全控制，`-no-window`无头 | ✅ 无屏幕概念 | CI/CD自动化（最佳） |
| 本机模拟器 | ✅ adb授权+autoGrant | ✅ 不会熄屏 | 开发调试 |
| 真机USB | ⚠️ adb可授大部分，厂商弹窗需特殊处理 | ✅ 已自动保持亮屏 | 兼容性测试 |

- 结论：**CI/CD场景优先用容器内模拟器**（Docker Android镜像如`budtmo/docker-android`），真机用于兼容性验证。
- 真机不能放容器内：USB设备需直接连宿主机，容器内即使`--privileged`也无法控制厂商弹窗。

#### L1 — 个人调试记忆（增强现有系统）

**现状**：`memory/store.py` 已有 `bug_patterns` + `fix_experiences` 表，但仅存储结构化字段。

**增强方向**：
1. **调试上下文快照**：记录 Bug 发生时的完整上下文（框架版本、OS、设备型号、蓝本步骤、错误堆栈）
2. **修复路径记录**：记录"从发现到修复"的完整链路（哪些文件改了、改了什么、为什么这样改）
3. **智能匹配提醒**：执行测试前，自动检索 memory.db 中的相似 Bug，在报告头部提示"上次你在这里遇到过类似问题，修复方案是 XXX"
4. **失败模式聚类**：同一选择器/同一页面反复失败 → 自动标记为"高风险区域"，测试时优先关注

#### L2 — 社区经验库（核心创新）

**工作流程**：
```
用户修复 Bug → 插件弹出提示：
  "你成功修复了一个 [Flutter/setState/U2无响应] 问题！
   要匿名分享给其他开发者吗？"
   [ 一键分享 ]  [ 不了 ]
       ↓
  匿名化处理：脱敏代码路径/变量名/业务逻辑
       ↓

       ↓
```

**匿名化规则（隐私安全第一）**：
- ✅ 保留：框架名、错误类型、修复策略、平台信息、选择器模式
- ❌ 脱除：项目名、文件路径、变量名、业务数据、用户信息、IP 地址
- 用户可在上传前预览匿名化后的内容，确认后才发送

**匹配引擎**：
- 按维度索引：`平台(web/android/miniprogram/desktop)` × `框架(flutter/react/vue)` × `错误类型(timeout/element_not_found/crash)` × `工具版本(appium/playwright)`
- 相似度计算：错误消息文本向量 + 上下文指纹匹配
- 推荐排序：解决率 > 相似度 > 时效性 > 点赞数

**技术方案**：
- 云端 API：`POST /api/community/share`（上传）+ `GET /api/community/suggest`（推荐）
- 存储：PostgreSQL + pgvector（向量检索）
- 缓存：热门经验本地 SQLite 缓存，离线可用
- 安全：所有通信 HTTPS，上传内容二次审核（AI 自动 + 人工抽检）

## v12.0 AI中枢决策层（架构规划）

> **愿景**：引擎从"AI当工具"升级为"AI当大脑"，像真人测试员一样观察→理解→决策→行动。

### 现状分析

当前架构：AI是被动工具，只在特定节点调用（截图分析、坐标定位、文字识别）。
```
蓝本步骤 → 执行操作 → 失败？→ AI帮忙找坐标
                              → 成功 → 下一步
```

**缺陷**：
- 不理解当前"处于什么界面状态"
- 不知道"操作是否真的生效了"
- 遇到意外情况（弹窗、加载中、界面变化）无法自主应对
- 蓝本写漏一步就连锁崩溃

### 目标架构

AI作为中枢决策大脑，每步执行后评估界面状态，自主决策下一步：
```
AI中枢 ← 截图+UI树+上一步结果
  → 理解："当前在什么界面？上一步操作生效了吗？"
  → 判断："出现了弹窗/加载中/错误页？"
  → 决策："继续执行蓝本 / 先处理弹窗 / 等待加载 / 标记Bug并跳过"
  → 行动：执行决策
```

### 分层策略（渐进式引入）

| 层级 | 名称 | 触发条件 | AI调用 | 额外耗时 |
|:---:|------|---------|:---:|:---:|
| **L0** | 无AI | 步骤成功 | 0 | 0s |
| **L1** | 弹窗自愈 | 步骤失败 | 1次 | 4-8s |
| **L2** | 上下文理解 | 失败+重试失败 | 2次 | 8-15s |
| **L3** | 全局决策 | 连续失败≥3步 | 3次 | 12-25s |

**关键原则**：步骤通过时零开销，只在失败时逐层升级AI介入深度。

### L1 弹窗自愈（v10.6+ 当前实现）

```
步骤失败
  → 截图
  → AI: "截图中是否有弹窗？OK/Cancel/Close按钮在哪？"
  → 有弹窗 → 点击关闭 → 重试原步骤
  → 无弹窗 → 标记Bug
```

### L2 上下文理解（v12.0-Phase1）

```
步骤失败 + L1重试仍失败
  → 截图 + UI树
  → AI: "当前界面是什么状态？与预期（蓝本描述）差异在哪？建议如何恢复？"
  → AI回答: "当前在设置页面，不是主页面。建议先导航回主页。"
  → 执行AI建议 → 重试原步骤
```

### L3 全局决策（v12.0-Phase2）

```
当前场景连续失败≥3步
  → 截图 + UI树 + 最近5步执行日志
  → AI: "场景已卡住，分析原因，给出建议：继续/跳过/重启应用"
  → AI回答: "应用进入了死循环状态，建议重启应用并跳到下一场景。"
  → 执行决策
```

| 5步失败（L1治愈） | 58步 | 91% | 5次 | 20-40s | ¥0.01 |

基于用户需求，确立插件UI设计的四大原则：
1. **项目隔离优先**：永不跨项目批跑
2. **平台透明优先**：显示全称平台，不用缩写
3. **运行前置检查优先**：先检查环境，再允许点击"开始测试"
4. **提示分层优先**：操作提示与内容提示分离，不互相覆盖

#### Phase 1: 项目隔离 + 平台全称显示

**实施内容**：
- 删除"全部项目"选项，项目下拉框只显示单个项目列表
- 项目显示格式：`{项目名}（{平台全称}）`
- 平台标签统一：web→Web、android→Android、ios→iOS、miniprogram→微信小程序、desktop→Windows桌面

**示例**：
- `FreshMart 生鲜超市（微信小程序）`
- `订单管理系统（Web）`
- `移动商城（Android）`

#### Phase 2: Tooltip分区（操作提示与内容说明分离）

**问题**：之前全局蓝本勾选时，一刀切覆盖整个item的tooltip，用户悬停蓝本内容区也只看到"已勾选全局蓝本"，看不到蓝本详情。

**解决方案**：
- 局部蓝本拆分为两个区域：`<span class="cbwrap">` + `<div class="info">`
- 复选框区域独立title：操作提示（"已启用全局蓝本，如需局部选择请先取消全局蓝本"）
- 内容区域独立title：蓝本详情（描述、场景数、步骤数、路径）
- 全局勾选时只修改复选框区title，内容区title保持不变

**用户体验提升**：
- 悬停复选框 → 看到操作引导
- 悬停蓝本标题/描述 → 始终看到蓝本功能说明

#### Phase 3: 平台前置检查（运行前环境检测+阻断逻辑）

**流程设计**：
1. 用户点击"▶ 运行选中蓝本"
2. 插件发送`platformPrecheck`消息到后端
3. 后端`_handlePlatformPrecheck`按平台执行检查：
   - **Android/iOS**：检测设备连接 → 未连接时阻断并提示 → 已连接时自动创建会话
   - **微信小程序**：检测开发者工具状态 → 未就绪时阻断并提示固定端口9420
   - **Web/Windows桌面**：直接通过检查
4. 检查通过后才发送`blueprintTest`或`blueprintBatchTest`请求

**新增API方法**（`engineClient.ts`）：
- `listMobileDevices()` - 列出已连接移动设备
- `listMobileSessions()` - 列出活动移动会话
- `createMobileSession()` - 创建移动会话
- `getMiniprogramDevtoolsStatus()` - 检查小程序开发者工具状态
- `startMobileBlueprintTest()` - 手机蓝本测试
- `startMiniprogramBlueprintTest()` - 小程序蓝本测试
- `startDesktopBlueprintTest()` - 桌面蓝本测试

### 产品设计文档：记忆系统与蓝本介入方案（2026-03-08讨论确定）

#### 一、记忆存储三层策略

经验和铁律的存储采用三层保险策略，确保关键信息不丢失：

| 层级 | 载体 | 特点 | 加载方式 |
|------|------|------|----------|
| **第1层：铁律层** | `.windsurf/rules.md` | 10条以内绝对铁律 | Windsurf每次会话**自动全量加载** |
| **第2层：详情层** | `开发备忘录.md` | 详细开发记录和经验 | AI按需`read_file`读取 |
| **第3层：辅助层** | Cascade持久记忆 | 跨项目经验参考 | 系统语义检索，每次3-5条 |

**设计原则**：
- 第1层保底（永不丢失），第2层详尽（想看就看），第3层辅助（锦上添花）
- 不过度依赖AI记忆系统的自动检索能力
- 关键铁律必须写在项目文件中，不能只存在AI记忆里

#### 二、跨项目经验传递格式

每条经验必须标注来源和可借鉴程度：

```
[来源: TestPilotAI] [仅供参考]
插件HTML中有JS引用的DOM元素，永远不要删除，只能用CSS隐藏。
原因：getElementById()返回null后，后续所有addEventListener()都会崩溃。
```

**规则**：
- 不需要AI自动区分"通用经验"和"项目特有经验"
- 只需要标注来源项目 + 统一写上"仅供参考"
- AI引用时说明："这是从XX项目得到的经验，你可以借鉴，但不保证适用"
- 大致匹配的经验都提供，由使用者自行判断是否采纳

#### 三、Cascade记忆系统的实际限制

| 能力 | 现状 |
|------|------|
| 存储 | 不会压缩已存内容本身 |
| 检索 | 每次会话按语义相关性返回3-5条，可能遗漏 |
| 去重 | 无法自动排除"已提供过的记忆" |
| 跨IDE | 不互通（Windsurf和Cursor各自独立） |

**应对方案**：铁律写在`.windsurf/rules.md`里，不依赖检索。

#### 四、蓝本生成介入方案

**核心原则**：关键节点触发，不做实时监控，Token消耗接近零。

| 触发节点 | 方式 | Token消耗 |
|----------|------|-----------|
| 插件首次安装/打开无蓝本项目 | UI醒目提示 + 两个按钮 | **零** |
| 引擎连接时 | 自动扫描蓝本文件 | **零** |
| 用户点"刷新🔄"按钮 | 重新扫描蓝本 | **零** |
| 编程AI修改代码时 | copilot-instructions规则提醒 | **低** |
| 用户主动生成蓝本 | 点按钮或复制提示词 | **中**（一次约2000-5000 token） |
| 测试报告底部 | 附带经验总结+蓝本维护提醒 | **零** |

**关于"用户编程完来不及生成蓝本就关了界面"**：
- 蓝本文件一旦生成就保存在项目目录，永不丢
- 下次打开插件自动扫描到
- copilot-instructions规则持续提醒编程AI维护蓝本
- 不需要担心断开后丢失

#### 五、iOS平台策略

| 操作系统 | iOS项目显示 | 行为 |
|---------|------------|------|
| **Windows** | 灰色不可选 | 温馨提示"iOS项目需要macOS环境才能运行测试" |
| **macOS** | 正常可选 | 设备检测 + 正常测试流程 |
| **Linux** | 灰色不可选 | 同Windows提示 |

**设计理由**：用户可能不懂技术限制（AI降低了编程门槛），需要友好提示而非直接隐藏。

#### 六、测试报告经验总结格式（待实现）

每次测试报告末尾附带：

```
═══════════════════════════════════════
📝 蓝本维护提醒
如果您修改了代码中的UI元素、业务逻辑或页面路由，
请更新蓝本文件：testpilot/testpilot.json

💡 经验总结（建议保存到 .windsurf/rules.md）
- [来源: 本项目] [仅供参考] 具体经验描述...
═══════════════════════════════════════
```

#### 七、插件开发铁律（从犯错中总结）

1. **永远不要删除HTML中有JS引用的DOM元素**（已犯2次：btnScanBp、btnCheckEngine）
   - 要隐藏用CSS `class="hidden"`
   - 修改前先搜索ID在JS中的引用
2. **插件改造用最小改动策略**，不要大规模重写
3. **每次改完必须编译测试**，确认基本功能（启动引擎、扫描蓝本、项目切换）正常

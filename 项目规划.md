## 一、项目愿景

打造一个AI驱动的自动化测试机器人，能够：
- 像人类一样操作应用界面（点击、输入、滑动）
- 自动发现Bug并生成详细报告
- 自动修复发现的问题并重新测试
- 让不懂技术/不懂特定领域的用户也能验证AI生成的代码

**核心价值**：补全AI编程工具链的"最后一公里"——验证。

## 二、已确认的核心设计要点

### 2.1 三AI交叉验证架构（创新点）

- **AI-A（编程AI）**：写出应用
- **AI-B（数据AI）**：独立生成测试数据（不同对话窗口，避免自我一致性偏差）
- **AI-C（测试AI）**：在沙箱中模拟人类操作，用B的数据测试A的应用
- 三个AI互相独立，不共享上下文，避免"自己考自己"

### 2.2 Docker沙箱环境（"桶"）

- 使用Docker Sandbox或AIO Sandbox作为隔离执行环境
- AI在沙箱内有完全权限，随意操作不影响宿主机系统
- 沙箱内集成：浏览器、Shell、文件系统、开发工具

**已确定方案：编程在本地，测试在桶里（方案A）**
- 用户照常用Cursor/Windsurf在本地编程
- 通过Docker Volume Mount将项目文件夹挂载到沙箱（如 `-v D:\Projects\MyApp:/workspace`）
- 文件实时同步，无需打包/部署步骤，AI全自动完成
- 沙箱内自动执行启动命令（如 `npm run dev`）运行应用
- 用户仅需首次配置：项目路径 + 启动命令

**关键约束**：
- 必须兼容市面上各种AI编程工具（Cursor、Copilot、Windsurf、TRAE等）
- 编程工具在本地运行，完全不受沙箱影响，零侵入

### 2.3 测试流程：用户审批制

1. 用户输入完整需求描述
2. AI生成测试脚本（含每步的预期结果）
3. 用户审批测试脚本（确认步骤和预期结果是否正确）
4. AI按脚本自动执行，逐步对比实际结果 vs 预期结果
5. 不一致的步骤记录为Bug
6. 生成完整测试报告，用户查看

**测试脚本格式示例**：
```
步骤1: 打开 /login 页面
  -> 预期: 显示登录表单，含用户名、密码、登录按钮

步骤2: 输入用户名"admin"，密码"123456"，点击登录
  -> 预期: 跳转到 /dashboard，显示欢迎信息
```

### 2.4 多模态分析：豆包视觉API

- 测试执行时录屏 + 关键帧提取（UI状态变化时截帧）
- 截帧发送给豆包视觉理解API进行分析
- 价格：0.003元/千token，1元可处理284张720P图片
- 每次完整30分钟测试的多模态分析成本约 0.75-2元

### 2.5 自动修复闭环（核心价值）

- 测试发现Bug -> 生成结构化Bug报告（含截图、实际vs预期、错误上下文）
- 交给编程AI自动修复 -> 修复后重新执行测试
- 通过 -> 标记"已自动修复，请确认"
- 未通过或无法修复 -> 标记"需人工介入"

### 2.6 记忆系统（SQLite本地存储）

- ~~原计划用Mem0，实际采用更轻量的SQLite方案~~
- 存储：测试历史、经验教训、页面指纹（`data/memory.db`）
- 零外部依赖，无需额外LLM后端
- 后续计划：历史记录定期压缩为经验总结

## 六、已决策事项（第三轮讨论）

### 6.1 多模态分析策略
- **每步截帧 + 事后视频录像作为补充**
- 每执行一个测试步骤截一帧，用于和预期结果精确对比
- 整个测试过程同时录屏，视频留存作为证据和回溯用
- 具体参数待开发中根据API调用成本动态调整

### 6.2 测试数据领域覆盖
- 当前无成熟的"领域测试数据生成API"可直接接入
- **第一阶段**：先用大模型自身知识生成测试数据
- **后续优化**：针对常见领域（财务、医疗、电商）准备提示词模板库
- **远期**：支持用户上传领域文档（如公司内部业务规则），AI参考生成更精准测试数据

### 6.3 测试遇Bug时的行为：分类处理
- **阻塞性Bug**（页面崩溃、无法进入下一步）→ 自动停下，等待修复后重测
- **非阻塞性Bug**（文字错误、布局偏移）→ 记录后继续执行
- 所有分类Bug跑完或跑不下去后，统一交给编程AI修复

### 6.4 技术选型（已确定）

**推荐技术栈**：

| 层 | 技术 | 选择理由 |
|----|------|---------|
| 核心引擎 | **Python + FastAPI** | AI生态最好，LangChain/Mem0/OpenAI SDK均Python优先 |
| 浏览器自动化 | **Playwright (Python)** | 支持Chromium/Firefox/WebKit，原生截图录屏，跨平台 |
| Docker管理 | **docker-py** | Python原生Docker SDK，成熟稳定 |
| 记忆系统 | **SQLite**（原计划Mem0） | 轻量、零依赖、本地存储 |
| IDE插件 | **TypeScript + VSCode Extension API** | Windsurf/VSCode通用，同一份代码 |
| 独立桌面应用 | **Tauri (Rust壳) + React + TypeScript** | 安装包3-10MB（Electron需50MB+），内存低，启动快 |
| 前端UI | **React + TailwindCSS** | IDE插件和桌面应用共享UI组件库 |
| 通信协议 | **HTTP/WebSocket (FastAPI ↔ 前端)** | 跨语言通信，前后端解耦 |

**Tauri vs Electron选型依据**：
| 维度 | Tauri | Electron |
|------|-------|----------|
| 安装包大小 | 3-10MB | 50MB+ |
| 内存占用 | 低（系统WebView） | 高（捆绑Chromium） |
| 启动速度 | 快 | 慢 |
| 安全性 | 高（Rust内存安全） | 中 |
| 跨平台一致性 | 中（WebView有差异） | 高 |
| 生态成熟度 | 中（增长快） | 高（老牌） |
| 更新维护 | 轻量，更新快 | 更新包大 |

选Tauri因为：桌面应用只是展示层（报告、配置、日志），重逻辑在Python引擎，需要应用尽量轻量，用户电脑同时还跑IDE+Docker+AI工具。

**系统架构**：
```
前端展示层
  ├── IDE插件: TypeScript + VSCode Extension API
  └── 桌面应用: Tauri (Rust) + React + TypeScript
      └── 共享UI组件库: React + TailwindCSS
              │
              │ HTTP / WebSocket
              ▼
核心引擎层: Python
  ├── FastAPI (HTTP服务)
  ├── AI调度: LangChain / 豆包API直调
  ├── 记忆系统: Mem0
  ├── Docker管理: docker-py
  └── 浏览器自动化: Playwright (Python)
```

**支持平台**：Windows、macOS、Linux（三平台，Tauri和Playwright均原生支持）

### 7.5 依赖管理原则（重要）
1. **锁定所有版本号** — 用 poetry.lock / package-lock.json 锁死版本
2. **使用虚拟环境** — Python用poetry或venv隔离
3. **最小化依赖** — 能不引入的库不引入
4. **提前验证兼容性** — 每加一个新依赖先在干净环境验证
5. **开发前先搭建并验证完整依赖链** — 确保所有库能一起安装运行

### 7.6 记忆系统：SQLite本地存储（v1.3已完整激活）
- ~~原计划用Mem0，但实际开发中发现需求较简单，用SQLite即可~~
- 当前实现：`src/memory/store.py`（5张表）+ `src/memory/compressor.py`（规则压缩器）
- 存储文件：`data/memory.db`（自动创建）
- 5张表：test_history、experiences、page_fingerprints、bug_patterns（v1.3）、fix_experiences（v1.3）
- 记忆提取：MemoryCompressor从BugReport/FixPlan固定字段直接提取，零AI成本
- 记忆检索：get_context_for_url()分层查询（Bug模式+修复经验+通过率趋势），≤500字符注入提示词
- 淘汰机制：cleanup()清理30天过期记录 + 同URL最多10条Bug模式
- 设计理念：格式化是前提→字段直接提取→零AI成本→经验=Bug发现+修复方案

## 八、已实现：智能修复模式（v1.3）

### 单一智能模式（取代原双模式设计）

经评估，"逐个修复"和"批量修复"各有优劣，最终采用**智能模式**——由 `SmartRepairDecider` 自动判断每个Bug的修复策略，用户无需选择：

**阻塞性Bug → 立即修复（IMMEDIATE）**
判断标准（满足任一即触发）：
1. Bug严重度为 CRITICAL
2. 步骤执行状态为 ERROR（浏览器操作失败）
3. 连续失败步骤数 ≥ 3
4. Bug涉及白屏/崩溃关键词

**非阻塞性Bug → 延迟批量修复（DEFERRED）**
- 计算错误、样式问题、文案错误等
- 攒到测试全部执行完后，一次性批量修复
- 自然去重（同类Bug不重复修）

**实现文件**
- `src/testing/smart_repair.py` — SmartRepairDecider 决策器
- `src/testing/blueprint_runner.py` — run() 方法集成 repair_callback
- 报告末尾自动附带智能修复统计

## 九、已决策：两种测试模式（核心架构）

### 模式一：蓝本模式（精确功能测试）—— 优先实现

**核心思路：编程AI生成代码时，必须同时输出测试蓝本**

蓝本 = 测试说明书，由编程AI（Windsurf/Cursor等）生成，内容包括：
- 页面有哪些元素、CSS选择器是什么
- 每个按钮/输入框的功能和预期效果
- 测试场景和步骤（点哪里、输什么、预期看到什么）

**蓝本格式示例（testpilot.json）：**
```json
{
  "app_name": "Todo待办应用",
  "pages": [
    {
      "url": "/",
      "title": "Todo 待办事项",
      "elements": {
        "输入框": "#todoInput",
        "添加按钮": "#addBtn",
        "待办列表": "#todoList"
      },
      "scenarios": [
        {
          "name": "添加待办",
          "steps": [
            {"action": "fill", "target": "#todoInput", "value": "测试任务"},
            {"action": "click", "target": "#addBtn"},
            {"expected": "列表中出现'测试任务'"}
          ]
        }
      ]
    }
  ]
}
```

**工作流程：**
1. 用户让编程AI写完功能后，要求它输出 `testpilot.json` 蓝本
2. TestPilot AI 读取蓝本，按精确选择器执行测试
3. 100%不会猜错选择器，专注于功能验证
4. 适用场景：开发中、提交前、功能验证

**为什么蓝本模式优先：**
- 当前"盲测"模式AI猜选择器经常猜错（实测验证，`data-testid` vs `#todoInput`）
- 用户最高频的需求是"确认功能是否正确"，蓝本模式最精准
- 编程AI完全有能力输出蓝本，只需要一段提示词引导

### 模式二：意外模式（AI自由探索测试）

**核心思路：AI自由操作页面，发现设计之外的Bug**

**方案A（优先实现）：批量截图分析**
- AI自由操作页面，连续执行10步
- 每步只截图+记录操作日志，不调AI分析（快、零成本）
- 10步结束后，10张截图+操作日志打包，一次性发AI分析
- 发现异常再做深度分析
- 一轮10步只调AI 1-2次，速度快、成本低

**方案B（备选）：视频分析模式**
- 录制几分钟操作视频
- 发给 Doubao-Seed-1.8 视频理解能力分析
- 模型支持低帧率超长视频理解
- 成本：输入0.8元/百万tokens，预估1分钟视频几分钱到一毛钱
- 方案A不够完善时补充使用

**使用场景：**
- 蓝本测试通过后，夜间/闲时放着跑
- 不需要用户盯着看，后台自动执行
- 发现的意外Bug汇总到报告中

### 实现优先级

1. **v1.x**：蓝本模式（解析 testpilot.json + 精确执行）
2. **v1.x**：意外模式方案A（批量截图分析）
3. **v2.0**：意外模式方案B（视频分析，视情况补充）

## 十、已决策：蓝本详细规范与异常上报机制

### 蓝本应该写多详细？

蓝本不只是"点哪里"，而是完整的**操作→等待→验证**三元组：

**每个步骤必须包含：**
1. **操作**：点击/输入/选择等
2. **等待**：操作后等多久再验证（`wait_after_ms`），处理异步加载/动画
3. **预期**：操作后应该看到什么变化

**蓝本增强字段（v1.x迭代）：**
```json
{
  "action": "click",
  "target": "#addBtn",
  "description": "点击添加按钮",
  "wait_after_ms": 500,
  "expected": "列表新增一行，显示刚才输入的文字；统计栏数字+1",
  "unexpected_is_bug": true
}
```

**智能输入：**
```json
{
  "action": "fill",
  "target": "#priceInput",
  "value": "auto:number:1-999",
  "description": "AI随机生成一个合理价格数字输入"
}
```

`auto:` 前缀表示让AI自主生成合理值：
- `auto:number:1-999` — 随机数字
- `auto:text:人名` — 随机生成一个合理的中文人名
- `auto:email` — 随机生成邮箱
- 固定值直接写字符串

**计算验证：**
```json
{
  "action": "assert_text",
  "target": "#totalPrice",
  "value": "=数量×单价",
  "description": "验证总价计算公式是否正确"
}
```

### 蓝本外异常的上报机制（通用异常检测）

**核心原则：蓝本没提到的异常也是异常，照样上报。**

编程AI写蓝本时可能有疏漏，不可能覆盖所有情况。
因此TestPilot在每步截图后，除了验证蓝本的`expected`，还自动做**通用异常检测**：

1. **页面崩溃/白屏** — 页面内容为空或出现错误页
2. **红色报错信息** — 页面上出现明显的错误提示
3. **控制台JS报错** — 浏览器控制台的error级别日志
4. **布局严重异常** — 元素重叠、溢出、消失等
5. **网络请求失败** — 关键API返回4xx/5xx

这些异常即使蓝本没写，也会作为**"蓝本外异常"**上报，标注为：
- 类型：`unexpected` 
- 说明：TestPilot自动检测，非蓝本预期范围
- 建议：请编程AI检查此异常是正常行为还是Bug

**用户/编程AI收到上报后自行判断**：
- 如果是Bug → 修复
- 如果是正常行为 → 下次在蓝本里标注"此为正常"（加入白名单）

#### v5.0 技术决策记录（2026-03-05讨论）

**问题0：手机蓝本格式和日志收集与 Web 测试是否一致？**
- **蓝本格式**：
  - 手机浏览器测试网页 → **完全相同**，CSS 选择器（`#addBtn`、`.btn`）在手机浏览器里同样精确，同一份 `testpilot.json` 直接复用，屏幕尺寸不影响选择器
  - 原生 App（APK）→ 需加 `"platform": "android"` 字段，目标改用 `resource-id`（`com.app:id/btn`）或 `accessibility-id`，属未来扩展
- **日志收集已实现 v5.0-E（已完成）**： `adb logcat` 守护线程 + Appium `get_log('browser')` + WebView 上下文自动切换，全部上线

**问题1：熄屏/锁屏时能否测试？**
- 结论：不能。UiAutomator2依赖屏幕UI元素树，熄屏后截图纯黑、操作失效。
- 方案：**自动保持亮屏（已实现）**。launch()时通过adb设置`stay_on_while_plugged_in=7`（充电时永不灭屏）+ `screen_off_timeout=2147483647`（超时最大值）+ `KEYCODE_WAKEUP`唤醒屏幕。close()时自动恢复原始值。用户零操作。

**问题2：权限弹窗如何处理（无人值守）？**
- 标准权限：Appium `autoGrantPermissions: true`（已实现），自动点击"允许"。
- adb预授权：`adb shell pm grant <pkg> <permission>`，可批量授权，无需人在场。
- 厂商定制弹窗（华为/小米/OPPO/vivo）：标准Appium无法覆盖，需维护厂商弹窗选择器库（v5.1计划）。
- 蓝本扩展：计划在蓝本中新增 `permissions` 字段，launch时自动adb授权。

**问题3：模拟器 vs 真机 vs 容器**
| 场景 | 权限 | 熄屏 | 适用 |
|------|------|------|------|
| 容器内模拟器 | ✅ 完全控制，`-no-window`无头 | ✅ 无屏幕概念 | CI/CD自动化（最佳） |
| 本机模拟器 | ✅ adb授权+autoGrant | ✅ 不会熄屏 | 开发调试 |
| 真机USB | ⚠️ adb可授大部分，厂商弹窗需特殊处理 | ✅ 已自动保持亮屏 | 兼容性测试 |

- 结论：**CI/CD场景优先用容器内模拟器**（Docker Android镜像如`budtmo/docker-android`），真机用于兼容性验证。
- 真机不能放容器内：USB设备需直接连宿主机，容器内即使`--privileged`也无法控制厂商弹窗。

##### A. MultiPlayerOrchestrator（多实例协调器）✅

##### B. 多人蓝本格式（Multi-Player Blueprint）
```json
{

    {"id": "player4", "platform": "android", "device": "设备serial"}
  ],

    {"player": "player1", "action": "tap", "selector": "#start-game"},
    {"sync": "all_players_ready", "timeout": 10},
    {"player": "player1", "action": "tap", "selector": ".card-wan-1"},

    {"parallel": [
      {"player": "player3", "action": "screenshot"},
      {"player": "player4", "action": "screenshot"}
    ]},
    {"assert_consistency": ["player1", "player2", "player3", "player4"], "check": "牌桌状态一致"}
  ]
}
```

##### C. 同步原语（Sync Primitives）✅
- [x] **`sync`（同步屏障）**：所有玩家到达此步骤后才继续，防止步骤乱序
- [x] **`wait_for`（条件等待）**：等待某个玩家的界面满足条件后继续
- [x] **`broadcast`（广播事件）**：通过EventBus实现玩家间事件通信
- [x] **`parallel`（并行执行）**：多个玩家同时执行操作（如同时截图对比）
- [x] **`assert_consistency`（一致性断言）**：ConsistencyChecker截图对比+差异报告

### v11.0-B 调试记忆 + 社区经验库（Debug Memory + Community Wisdom）

> **愿景**：让每一次修复，都成为未来的捷径。
> 个人记忆 + 社区智慧 = 越用越聪明的 AI 编程助手。

#### 核心理念

用户在项目中踩过的坑、AI 帮修过的 Bug，都会被自动记录为"调试记忆"。
- **避免自己重复犯错**：同类问题自动提醒
- **匿名分享给全球开发者**：一键上传，全凭自愿
- **智能推荐"别人是怎么修的"**：精确到框架和平台

#### 三层架构

| 层级 | 名称 | 范围 | 存储 | 说明 |
|:---:|------|------|------|------|
| **L1** | 个人调试记忆 | 本地私有 | SQLite memory.db | 自动记录每次踩坑+修复过程，零配置 |
| **L2** | 社区经验库 | 全球云端 | 云数据库（PostgreSQL） | 匿名共享解决方案，按框架/平台/错误类型索引 |
| **L3** | 成就勋章 | 账号绑定 | 云端用户表 | 贡献者获得专属成就，形成社区认同和荣誉感 |

#### L1 — 个人调试记忆（增强现有系统）

**现状**：`memory/store.py` 已有 `bug_patterns` + `fix_experiences` 表，但仅存储结构化字段。

**增强方向**：
1. **调试上下文快照**：记录 Bug 发生时的完整上下文（框架版本、OS、设备型号、蓝本步骤、错误堆栈）
2. **修复路径记录**：记录"从发现到修复"的完整链路（哪些文件改了、改了什么、为什么这样改）
3. **智能匹配提醒**：执行测试前，自动检索 memory.db 中的相似 Bug，在报告头部提示"上次你在这里遇到过类似问题，修复方案是 XXX"
4. **失败模式聚类**：同一选择器/同一页面反复失败 → 自动标记为"高风险区域"，测试时优先关注

**实现文件**：
- `src/memory/debug_memory.py` — 调试记忆增强模块
- `src/memory/store.py` — 新增 `debug_snapshots` 表（上下文快照+修复路径）

#### L2 — 社区经验库（核心创新）

**工作流程**：
```
用户修复 Bug → 插件弹出提示：
  "你成功修复了一个 [Flutter/setState/U2无响应] 问题！
   要匿名分享给其他开发者吗？"
   [ 一键分享 ]  [ 不了 ]
       ↓
  匿名化处理：脱敏代码路径/变量名/业务逻辑
       ↓
  上传到云端经验库
       ↓
  其他用户遇到类似问题时：
  "💡 社区建议：有 23 位开发者遇到过类似问题，最有效的方案是..."
```

**匿名化规则（隐私安全第一）**：
- ✅ 保留：框架名、错误类型、修复策略、平台信息、选择器模式
- ❌ 脱除：项目名、文件路径、变量名、业务数据、用户信息、IP 地址
- 用户可在上传前预览匿名化后的内容，确认后才发送

**匹配引擎**：
- 按维度索引：`平台(web/android/miniprogram/desktop)` × `框架(flutter/react/vue)` × `错误类型(timeout/element_not_found/crash)` × `工具版本(appium/playwright)`
- 相似度计算：错误消息文本向量 + 上下文指纹匹配
- 推荐排序：解决率 > 相似度 > 时效性 > 点赞数

**技术方案**：
- 云端 API：`POST /api/community/share`（上传）+ `GET /api/community/suggest`（推荐）
- 存储：PostgreSQL + pgvector（向量检索）
- 缓存：热门经验本地 SQLite 缓存，离线可用
- 安全：所有通信 HTTPS，上传内容二次审核（AI 自动 + 人工抽检）

**免费开放**：
- 所有用户免费使用社区经验库（查看推荐）
- 免费用户也能上传贡献，不限次数
- 不需要注册也能查看推荐（匿名访问）

#### 路线图调整说明（2026-03-05讨论决定）

**核心策略：商业化优先，平台覆盖按需**

1. **为什么团队功能提前到v6.0？**

   - Win/Mac/Linux三平台同时做工作量巨大，且市场需求小于Web+手机
   - 目标用户画像：个人开发者、AI新手、小型工作室 → Windows占比最高（~62%）
   - 先做Windows + Electron，覆盖绝大多数目标用户，Mac/Linux根据反馈决定

3. **为什么加入v5.2报告增强？**
   - 用户感知价值高（图表比数字直观），工作量小（~1周）
   - 为商业化做铺垫：好看的报告 = 更容易说服团队付费

4. **v8.0+ 远期规划不是不做，是按需做**
   - iOS需要Mac环境，成本高，等有明确需求再投入
   - Mac/Linux桌面测试等用户群达到一定量级再做
   - ~~微信小程序属于锦上添花~~ → **已实战验证（2026-03-07盲测v5: 12场景5Bug/0误报）**
   - 高级手势等属于锦上添花，优先级低于商业化

### v14.0-C：Bug自动上传经验库

> **目标**：测试完成后，Bug自动匿名化上传到云端经验库，其他用户遇到类似问题时能看到解决方案

#### Step 1：后端Bug上传API

**新增端点**：`POST /api/v1/community/auto-share`（区别于手动share）

```python
# src/community/routes.py 新增
@router.post("/auto-share", tags=["community"])
async def auto_share_bugs(
    req: AutoShareRequest,  # bugs: list[BugData], platform, framework
    user: User = Depends(get_current_user),
    db: Session = Depends(get_db)
):
    """批量匿名化上传Bug到社区经验库"""
    results = []
    for bug in req.bugs:
        # 1. 匿名化处理（复用已有 anonymizer.py）
        anon = anonymize_experience({
            "title": bug.title,
            "platform": req.platform,
            "problem_desc": bug.description,
            "solution_desc": bug.fix_suggestion or "待解决",
            "error_type": bug.category,
        })
        # 2. 创建经验记录
        exp = service.create_experience(db, user.id, anon)
        results.append({"id": exp.id, "title": exp.title})
    return {"shared": len(results), "experiences": results}
```

## v12.0 AI中枢决策层（✅ 已完成 v12.0-12.2）

> **愿景**：引擎从"AI当工具"升级为"AI当大脑"，像真人测试员一样观察→理解→决策→行动。

### 现状分析

当前架构：AI是被动工具，只在特定节点调用（截图分析、坐标定位、文字识别）。
```
蓝本步骤 → 执行操作 → 失败？→ AI帮忙找坐标
                              → 成功 → 下一步
```

**缺陷**：
- 不理解当前"处于什么界面状态"
- 不知道"操作是否真的生效了"
- 遇到意外情况（弹窗、加载中、界面变化）无法自主应对
- 蓝本写漏一步就连锁崩溃

### 目标架构

AI作为中枢决策大脑，每步执行后评估界面状态，自主决策下一步：
```
AI中枢 ← 截图+UI树+上一步结果
  → 理解："当前在什么界面？上一步操作生效了吗？"
  → 判断："出现了弹窗/加载中/错误页？"
  → 决策："继续执行蓝本 / 先处理弹窗 / 等待加载 / 标记Bug并跳过"
  → 行动：执行决策
```

### 分层策略（渐进式引入）

| 层级 | 名称 | 触发条件 | AI调用 | 额外耗时 |
|:---:|------|---------|:---:|:---:|
| **L0** | 无AI | 步骤成功 | 0 | 0s |
| **L1** | 弹窗自愈 | 步骤失败 | 1次 | 4-8s |
| **L2** | 上下文理解 | 失败+重试失败 | 2次 | 8-15s |
| **L3** | 全局决策 | 连续失败≥3步 | 3次 | 12-25s |

**关键原则**：步骤通过时零开销，只在失败时逐层升级AI介入深度。

### L1 弹窗自愈（v10.6+ 当前实现）

```
步骤失败
  → 截图
  → AI: "截图中是否有弹窗？OK/Cancel/Close按钮在哪？"
  → 有弹窗 → 点击关闭 → 重试原步骤
  → 无弹窗 → 标记Bug
```

### L2 上下文理解（v12.0-Phase1）

```
步骤失败 + L1重试仍失败
  → 截图 + UI树
  → AI: "当前界面是什么状态？与预期（蓝本描述）差异在哪？建议如何恢复？"
  → AI回答: "当前在设置页面，不是主页面。建议先导航回主页。"
  → 执行AI建议 → 重试原步骤
```

### L3 全局决策（v12.0-Phase2）

```
当前场景连续失败≥3步
  → 截图 + UI树 + 最近5步执行日志
  → AI: "场景已卡住，分析原因，给出建议：继续/跳过/重启应用"
  → AI回答: "应用进入了死循环状态，建议重启应用并跳到下一场景。"
  → 执行决策
```

### 成本估算

| 场景 | 步骤 | 通过率 | 额外AI调用 | 额外耗时 | 额外成本 |
|------|------|:---:|:---:|:---:|:---:|
| 全部通过 | 58步 | 100% | 0 | 0s | ¥0 |
| 5步失败（L1治愈） | 58步 | 91% | 5次 | 20-40s | ¥0.01 |
| 5步失败（L2介入） | 58步 | 91% | 10次 | 60-90s | ¥0.02 |
| 全场景崩溃（L3兜底） | 58步 | 50% | 15次 | 120-180s | ¥0.05 |

**结论**：即使最坏情况（L3全面介入），额外成本不到5分钱，完全可接受。

### 实现路线

1. **v10.6+**：L1弹窗自愈（当前阶段）— `_execute_step` 失败后截图→AI检测弹窗→关闭→重试
2. **v12.0-Phase1**：L2上下文理解 — 引入"界面状态描述"概念，AI对比蓝本预期vs实际
3. **v12.0-Phase2**：L3全局决策 — 场景级智能跳转，应用重启恢复

#### 1. 项目选择器（核心交互改动）

**位置**：插件顶部，引擎状态栏右侧，新增一个项目下拉框。

**UI示意**：
```
┌─────────────────────────────────────────┐
│ 🔌 引擎已连接   [FreshMart 💬 ▾] [🔄] │  ← 项目下拉框 + 刷新按钮
├─────────────────────────────────────────┤

├─────────────────────────────────────────┤
│        [ 🚀 开始蓝本测试 ]              │  ← 自动走小程序引擎
└─────────────────────────────────────────┘
```

**三种状态**：

| workspace情况 | 下拉框行为 | 蓝本列表 |
|--------------|-----------|---------|
| **无项目**（没有testpilot.json） | 显示"暂无测试项目"灰色不可选 | 显示引导文字："请在项目中创建蓝本，或点击「📋 复制蓝本生成提示词」让编程AI生成" |
| **单项目** | 自动锁定，不可展开，直接显示项目名+平台标签 | 该项目的蓝本 |
| **多项目** | 可展开选择，列出所有项目 | 跟随当前选中项目自动过滤 |

**下拉框显示格式**：`项目名 + 平台徽章`
- 例：`FreshMart 生鲜超市 💬` / `BuggyShop 测试商城 🌐` / `MyApp 桌面应用 🖥️`
- 项目名来自蓝本的`app_name`字段
- 平台徽章自动从蓝本`platform`字段读取，用户不需要选平台
- 如同一项目下有多个蓝本且platform不同，取第一个蓝本的platform显示

**项目识别规则**：扫描workspace中所有含`testpilot.json`或`testpilot/`目录的项目文件夹，每个文件夹 = 一个项目。

---

#### 2. 扫描时机（不轮询，按需触发）

| 触发时机 | 说明 |
|---------|------|
| **插件激活时** | 打开IDE / 切换workspace → 自动扫描一次 |
| **用户点刷新 🔄** | 下拉框旁的刷新按钮，手动触发重新扫描 |
| **引擎连接成功时** | 连接引擎后自动重新扫描一次 |

**不做定时轮询**，原因：
- 浪费资源，大部分时间workspace不变
- 用户正在编辑蓝本时突然刷新会干扰
- 写完蓝本后点一下🔄就行，操作节奏由用户控制

---

#### 3. 平台自动路由（用户无感）

用户点"开始测试"后，插件根据蓝本的`platform`字段**自动选择执行引擎**：

| platform字段 | 执行路径 | API端点 |
|-------------|---------|--------|
| `web`（默认） | Playwright浏览器引擎 | `POST /api/v1/test/blueprint` |
| `miniprogram` | automator桥接服务器 | 自动启动`cli auto --auto-port 9420` + 桥接 |
| `android` | Appium真机引擎 | `POST /api/v1/test/mobile-blueprint` |
| `desktop` | Win32 UI Automation | 桌面控制器API |

**用户完全不需要知道"平台"概念**——他只选"我要测哪个项目"，剩下全自动。

1. 检测微信开发者工具是否已启动
2. 未启动 → 执行`cli open --project <path>`打开
3. 执行`cli auto --project <path> --auto-port 9420`启动自动化
4. 通过automator桥接服务器执行蓝本
5. 导航全部用`evaluate(wx.xxx())`（绝不用SDK方法）
6. 元素查找带重试机制（1.5秒间隔，最多3次）

### 产品设计文档：记忆系统与蓝本介入方案（2026-03-08讨论确定）

#### 一、记忆存储三层策略

经验和铁律的存储采用三层保险策略，确保关键信息不丢失：

| 层级 | 载体 | 特点 | 加载方式 |
|------|------|------|----------|
| **第1层：铁律层** | `.windsurf/rules.md` | 10条以内绝对铁律 | Windsurf每次会话**自动全量加载** |
| **第2层：详情层** | `开发备忘录.md` | 详细开发记录和经验 | AI按需`read_file`读取 |
| **第3层：辅助层** | Cascade持久记忆 | 跨项目经验参考 | 系统语义检索，每次3-5条 |

**设计原则**：
- 第1层保底（永不丢失），第2层详尽（想看就看），第3层辅助（锦上添花）
- 不过度依赖AI记忆系统的自动检索能力
- 关键铁律必须写在项目文件中，不能只存在AI记忆里

#### 二、跨项目经验传递格式

每条经验必须标注来源和可借鉴程度：

```
[来源: TestPilotAI] [仅供参考]
插件HTML中有JS引用的DOM元素，永远不要删除，只能用CSS隐藏。
原因：getElementById()返回null后，后续所有addEventListener()都会崩溃。
```

**规则**：
- 不需要AI自动区分"通用经验"和"项目特有经验"
- 只需要标注来源项目 + 统一写上"仅供参考"
- AI引用时说明："这是从XX项目得到的经验，你可以借鉴，但不保证适用"
- 大致匹配的经验都提供，由使用者自行判断是否采纳

#### 三、Cascade记忆系统的实际限制

| 能力 | 现状 |
|------|------|
| 存储 | 不会压缩已存内容本身 |
| 检索 | 每次会话按语义相关性返回3-5条，可能遗漏 |
| 去重 | 无法自动排除"已提供过的记忆" |
| 跨IDE | 不互通（Windsurf和Cursor各自独立） |

**应对方案**：铁律写在`.windsurf/rules.md`里，不依赖检索。

#### 四、蓝本生成介入方案

**核心原则**：关键节点触发，不做实时监控，Token消耗接近零。

| 触发节点 | 方式 | Token消耗 |
|----------|------|-----------|
| 插件首次安装/打开无蓝本项目 | UI醒目提示 + 两个按钮 | **零** |
| 引擎连接时 | 自动扫描蓝本文件 | **零** |
| 用户点"刷新🔄"按钮 | 重新扫描蓝本 | **零** |
| 编程AI修改代码时 | copilot-instructions规则提醒 | **低** |
| 用户主动生成蓝本 | 点按钮或复制提示词 | **中**（一次约2000-5000 token） |
| 测试报告底部 | 附带经验总结+蓝本维护提醒 | **零** |

**关于"用户编程完来不及生成蓝本就关了界面"**：
- 蓝本文件一旦生成就保存在项目目录，永不丢
- 下次打开插件自动扫描到
- copilot-instructions规则持续提醒编程AI维护蓝本
- 不需要担心断开后丢失

#### 五、iOS平台策略

| 操作系统 | iOS项目显示 | 行为 |
|---------|------------|------|
| **Windows** | 灰色不可选 | 温馨提示"iOS项目需要macOS环境才能运行测试" |
| **macOS** | 正常可选 | 设备检测 + 正常测试流程 |
| **Linux** | 灰色不可选 | 同Windows提示 |

**设计理由**：用户可能不懂技术限制（AI降低了编程门槛），需要友好提示而非直接隐藏。

#### 六、测试报告经验总结格式（待实现）

每次测试报告末尾附带：

```
═══════════════════════════════════════
📝 蓝本维护提醒
如果您修改了代码中的UI元素、业务逻辑或页面路由，
请更新蓝本文件：testpilot/testpilot.json

💡 经验总结（建议保存到 .windsurf/rules.md）
- [来源: 本项目] [仅供参考] 具体经验描述...
═══════════════════════════════════════
```

#### 七、插件开发铁律（从犯错中总结）

1. **永远不要删除HTML中有JS引用的DOM元素**（已犯2次：btnScanBp、btnCheckEngine）
   - 要隐藏用CSS `class="hidden"`
   - 修改前先搜索ID在JS中的引用
2. **插件改造用最小改动策略**，不要大规模重写
3. **每次改完必须编译测试**，确认基本功能（启动引擎、扫描蓝本、项目切换）正常

```python
# 表1: shared_experiences — 社区分享的匿名经验
class SharedExperience(Base):
    __tablename__ = "shared_experiences"
    id              # 主键
    user_id         # 外键 → users.id
    title           # 标题（如"Flutter setState导致U2无响应"）
    platform        # web/android/ios/miniprogram/desktop
    framework       # flutter/react/vue/tkinter/pyqt5/wxml...
    error_type      # timeout/element_not_found/crash/assertion/layout...
    problem_desc    # 匿名化后的问题描述
    solution_desc   # 匿名化后的解决方案
    code_snippet    # 匿名化后的代码片段（可选）
    root_cause      # 根因分类
    tags            # JSON标签数组 ["flutter","appium","uiautomator2"]
    tool_versions   # JSON {"appium":"3.2.0","playwright":"1.x"}
    difficulty      # easy/medium/hard（来自anonymizer）
    share_score     # 分享价值评分（来自calc_share_score）
    view_count      # 浏览次数
    upvote_count    # 点赞数
    adoption_count  # 被采纳次数
    status          # active/hidden/flagged/deleted
    created_at / updated_at

# 表2: experience_votes — 投票/采纳记录
class ExperienceVote(Base):
    __tablename__ = "experience_votes"
    id, experience_id(FK), user_id(FK)
    vote_type       # upvote/downvote/adopt
    created_at
    # 唯一索引: (experience_id, user_id, vote_type)

# 表3: user_badges — 用户勋章
class UserBadge(Base):
    __tablename__ = "user_badges"
    id, user_id(FK)
    badge_type      # first_share/helpful_10/flutter_expert/...
    badge_name      # 显示名称
    earned_at

# 表4: user_profiles — 用户扩展资料
class UserProfile(Base):
    __tablename__ = "user_profiles"
    user_id(FK, PK)
    display_name    # 昵称
    avatar_url      # 头像URL
    bio             # 一句话简介
    expertise_tags  # JSON 自动生成的专业标签
    stats_cache     # JSON 统计缓存（分享数/采纳数/帮助人数）
    notification_settings  # JSON 通知偏好

# 表5: debug_snapshots — 调试上下文快照（个人私有）
class DebugSnapshot(Base):
    __tablename__ = "debug_snapshots"
    id, user_id(FK), project_id(FK, nullable)
    platform, framework
    error_message, error_stack
    context         # JSON 完整上下文（步骤/选择器/蓝本片段）
    fix_description # 修复方案描述
    fix_files       # JSON 修改了哪些文件
    resolved        # bool
    shared_experience_id  # FK nullable → 是否已分享到社区
    created_at, resolved_at

# 表6: api_keys — 第三方API密钥（商业化）
class APIKey(Base):
    __tablename__ = "api_keys"
    id, user_id(FK)
    key_hash        # bcrypt哈希（不存明文）
    key_prefix      # 前8位明文（用于显示 tp_xxxx****）
    name            # 密钥名称
    permissions     # JSON ["read","test","blueprint"]
    last_used_at, expires_at, created_at
```

### v14.0-B：TOTP 双因素认证（计划中）

**技术方案**：
- 后端：`pyotp` 库（无需第三方服务，零费用、永久免费）
- 用户绑定：扫描二维码 → Google Authenticator / Microsoft Authenticator / Authy 均可
- 前端：登录后弹出 6 位验证码输入框
- 数据库：User 表增加 `totp_secret`（加密存储）和 `totp_enabled` 字段

**为什么选标准 TOTP 而不是腾讯虚拟MFA**：
- 腾讯虚拟MFA：底层同样是 TOTP，但用的是腾讯云企业管理场景，不适合对外SaaS产品
- 标准 TOTP：与任意验证器App兼容（全球1亿+用户在用），无供应商绑定，无平台费用
- 开发工作量：两者相同，标准TOTP用 `pyotp` 20行代码搞定

**实施步骤**：
1. `src/auth/models.py` — User 表增加 `totp_secret`, `totp_enabled` 两个字段
2. `alembic/versions/` — 创建迁移文件
3. `src/auth/service.py` — 增加 `generate_totp_secret()`, `verify_totp()`, `enable_totp()` 方法
4. `src/auth/routes.py` — 增加 3 个端点：`/auth/2fa/setup`、`/auth/2fa/verify`、`/auth/2fa/disable`
5. `web/src/pages/LoginPage.jsx` — 登录后检测 `require_2fa` 标志，弹出验证码步骤
6. `web/src/pages/DashboardPage.jsx` — 安全设置区：开启/关闭 2FA

**安全设计**：
- TOTP 密钥加密存储（AES-256，密钥来自环境变量 `TP_TOTP_ENCRYPT_KEY`）
- 每个验证码只能使用一次（防重放：记录上次使用的 counter）
- 允许前后30秒时间窗口（容忍手机时钟偏差）

```
    │                              │   ↓
    │                              │   调用豆包/OpenAI API
    │                              │   (API Key 在服务器环境变量)
    │◄── JSON 结果 ────────────────│
```

**二级域名方案（服务器 IP 变化时只改 DNS，前端代码零改动）**：

| 域名 | 指向 | 用途 |
|------|------|------|
| `testpilot.xinzaoai.com` | Vercel / Nginx | 官网静态内容 |
| `api.testpilot.xinzaoai.com` | 境外服务器 IP | 后端 FastAPI 服务 |
| `community.testpilot.xinzaoai.com` | 同后端服务器 | 社区 API（可合并） |

**API Key 安全规则（不允许例外）**：
1. `豆包/OpenAI API Key` 只能存在服务器 `.env` 文件或环境变量中
2. 前端（插件/网站）**绝对不能**持有或传输 AI API Key
3. BYOK 模式下，用户 Key 加密后存 VS Code SecretStorage，调用时在 HTTPS 请求头里传给后端，后端用完即弃，**不落库、不打日志**
4. 生产环境所有 API 端点必须有 JWT 鉴权（`/health` 等探活接口除外）

**服务器选型建议**：
- 境外（必须）：用于接收 Paddle Webhook + 服务海外用户，推荐 DigitalOcean $6/月（有 Nginx + Certbot 教程完善）

### 13.8 插件登录与身份验证方案（v14.2 必须实现）

> **核心要求**：插件启动时必须验证用户已登录，未登录的用户无法使用任何测试功能。

#### 用户流程

```
首次安装插件
    │
    ▼
侧边栏显示"请登录"界面
（邮箱 + 密码 输入框，或"点击跳转官网注册"按钮）
    │
    ▼
用户输入账号密码 → 插件发请求到 api.testpilot.xinzaoai.com/api/v1/auth/login
    │
    ▼
服务器验证通过 → 返回 JWT Token（access_token）
    │
    ▼
插件用 VS Code SecretStorage 加密保存 Token（非明文，用户看不到）
    │
    ▼
此后所有 API 请求自动在 Header 携带：Authorization: Bearer <token>
    │
    ▼
Token 24小时有效，过期后自动跳回登录界面
```

#### 技术实现要点

| 要点 | 说明 |
|------|------|
| Token 存储 | `context.secrets.store('tp_token', token)` — VS Code 内置加密存储，比 globalState 安全 |
| 未登录拦截 | 插件所有命令（开始测试/查看报告等）执行前先检查 Token，无 Token 直接弹登录 |
| Token 过期处理 | API 返回 401 时，自动清除本地 Token 并弹出重新登录提示 |
| 记住登录状态 | SecretStorage 持久化，重启 VS Code 不需要重新登录 |
| 注册入口 | 插件内"还没有账号？前往官网注册"按钮，用 `vscode.env.openExternal()` 打开官网 |

#### 插件侧边栏状态机

```
未登录态                    已登录态（免费）            已登录态（付费）
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ TestPilot AI    │         │ TestPilot AI    │         │ TestPilot AI    │
│                 │  登录   │ 👤 user@qq.com  │  充值   │ 👤 user@qq.com  │
│ [邮箱输入框]    │ ──────► │ 积分: 50        │ ──────► │ 积分: 500       │
│ [密码输入框]    │         │                 │         │                 │
│ [登录按钮]      │         │ [开始测试] ✅   │         │ [开始测试] ✅   │
│ [前往官网注册]  │         │ [查看报告] ✅   │         │ [查看报告] ✅   │
└─────────────────┘         └─────────────────┘         └─────────────────┘
```

## 十七、一键安装方案（上架核心卡点）

> **决策日期**：2026-03-30
> **优先级**：🔴 上架前必须完成，是转化率的最大障碍

### 问题定义

当前用户安装后需要：
1. 安装 Python 3.10+
2. 创建 venv 虚拟环境
3. `pip install -r requirements.txt`（含 Playwright、FastAPI 等重型依赖）
4. `playwright install chromium`
5. 手动运行 `python main.py` 启动引擎

这5个步骤对非技术用户而言是不可逾越的门槛。

### 技术约束分析

引擎**必须跑在用户本地**（不能纯云端），原因：
- Playwright 需要访问用户的被测应用（`localhost:3000` 等）
- 云端引擎无法穿透用户的内网/本地端口
- 截图、浏览器操作须在用户本地的 Chromium 中执行

因此，目标是：**用户只需装 VSIX，引擎自动就位，无需手动操作任何命令行**。

### 选型决策：PyInstaller 预打包二进制（主方案）

这是 VS Code 生态的标准做法（Pylance、Rust Analyzer、clangd 均采用此模式）：
- 引擎用 PyInstaller 打成单文件可执行程序（Windows: `.exe`，Mac/Linux: 无扩展名可执行文件）
- 二进制内含 Python 解释器 + 所有依赖（含 Playwright 的 Chromium）
- 插件首次激活时自动下载对应平台的二进制，无需用户操作
- 后续启动引擎只需 `spawn(engineBinaryPath)`

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| PyInstaller 预打包 | 零依赖，用户体验最好 | 二进制大（~150MB），首次下载等待 | ✅ **主方案** |
| 一键安装脚本 | 不需要维护大文件 | 仍需用户有 Python；企业网络可能阻断 pip | ❌ 不够彻底 |
| Docker | 彻底隔离，可配 TeamCity | 需用户有 Docker Desktop；Docker 启动慢 | ⚠️ 远期可作补充 |
| 纯云端引擎 | 零本地安装 | 无法访问 localhost，核心功能受限 | ❌ 技术上不可行 |

### 实施计划（分4步）

#### Step 1：创建 GitHub Actions 构建流水线

**文件**：`.github/workflows/build-engine.yml`

触发条件：打 `engine-v*` 标签时自动打包三平台

```yaml
# 三个 job，分别在对应平台的 runner 上构建
jobs:
  build-windows:  # runs-on: windows-latest
  build-macos:    # runs-on: macos-latest
  build-linux:    # runs-on: ubuntu-latest
```

**构建命令**（每个平台相同逻辑）：
```bash
pip install pyinstaller
pip install -r requirements.txt
playwright install chromium   # 把 Chromium 也打进去
pyinstaller main.py \
```

#### Step 2：插件端「引擎管理器」模块

**新文件**：`extension/src/engineManager.ts`

核心职责：
1. **检测**：启动时检查本地是否有引擎二进制（路径：`~/.testpilot/engine/`）
2. **下载**：若无或版本过旧，自动从 GitHub Releases 下载对应平台二进制
3. **权限**：Mac/Linux 需 `chmod +x`
4. **启动**：`spawn(engineBinaryPath, ['--port', '8000'])`
5. **版本检查**：定期检查 GitHub Releases 是否有新版本，提示用户更新

```typescript
// 核心接口
class EngineManager {
  async ensureEngineReady(): Promise<void>   // 插件激活时调用
  async downloadEngine(version: string): Promise<void>  // 带进度条

  async checkForUpdates(): Promise<boolean>
```

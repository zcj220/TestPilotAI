## 六、测试模式使用策略（重要！务必读懂）

### 6.1 通过率的正确理解

**通过率低 ≠ TestPilot 有问题，而是被测应用有 Bug！**

| 通过率 | 含义 |
|--------|------|
| 83%（30/36步通过） | TestPilot 正常工作，在被测应用中发现了 6 处问题 |
| 100% | 应用功能完全符合蓝本预期，没有 Bug |
| 0%（引擎报错） | 说明引擎/浏览器/环境有问题，需要先排查服务 |

**类比**：体检报告里检出 3 个指标异常，不是体检机器坏了——是身体有问题，体检机器工作越精确越好。

### 6.2 蓝本模式（当前主要模式）

**蓝本模式是日常开发的核心，现阶段几乎只用这个。**

特点：
- 按 `testpilot.json` 里的 CSS 选择器精确操作，不靠 AI 猜页面结构
- 速度快：36 步仅 87 秒
- 结果可重复：每次执行路径完全一致
- 适合：开发中功能验证、提交前回归、CI/CD 自动触发

启动流程：
```powershell
# 1. 启动引擎（新窗口）
Start-Process powershell -ArgumentList '-NoExit','-Command','cd D:\Projects\TestPilotAI; poetry run python main.py'

# 2. 启动浏览器（一次性，重启引擎后需重新执行）
Invoke-RestMethod http://localhost:8900/api/v1/browser/launch -Method POST -Body '{}' -ContentType 'application/json'

# 3. 启动被测应用（例如 BuggyShop）
Start-Process powershell -ArgumentList '-NoExit','-Command','cd D:\Projects\TestPilotAI\test-app; python -m http.server 8080'

# 4. 跑蓝本测试
poetry run python cli.py run -b test-app\testpilot.json -f json -o data\report.json
```

### 6.3 探索模式（暂缓使用）

**探索模式目前不是优先项，有以下限制：**
- AI 自由操作页面，每步都要截图+AI 分析，速度慢（分钟级）
- 结果不确定性高，难以回归对比
- **适合场景**：夜间/闲时让它在后台跑，发现蓝本之外的意外 Bug
- **目前建议**：优先把蓝本写全，覆盖所有核心场景。探索模式留到后续版本或夜间挂机用

### 6.4 真机蓝本测试——屏幕尺寸与选择器

**核心结论：蓝本模式天生支持多机型，不存在屏幕尺寸问题。**

| 测试类型 | 蓝本写法 | 与屏幕尺寸关系 |
|---------|---------|-------------|
| 手机浏览器打开网页 | CSS 选择器（`#addBtn`、`.submit-btn`） | **无关**，浏览器负责布局，选择器永远精确 |
| 原生 App（APK） | resource-id（`com.app:id/btn_submit`）或 accessibility-id | **无关**，元素 ID 不随屏幕变化 |
| 滑动（swipe）操作 | 使用百分比坐标（如"从屏幕 50%位置滑到 20%"）| 需要做适配，用百分比而非固定像素 |

**结论**：同一份蓝本在 5 寸小屏手机和 7 寸平板上跑，结果完全一致，不需要为不同设备写多份蓝本。

**v5.0-E 已完成（2026-03-05）**：新增独立 `POST /api/v1/test/mobile-blueprint` 端点（传 `mobile_session_id`），由 `MobileBlueprintRunner` 通过 Appium + WebView 上下文执行蓝本，同一份 `testpilot.json` 真机可直接运行。

### 6.5 手机蓝本格式 vs 日志收集——详细对比

#### 蓝本格式（testpilot.json）

| 场景 | 选择器格式 | 是否需要新字段 |
|------|-----------|--------------|
| 手机浏览器测 Web 网页 | CSS 选择器（`#addBtn`、`.btn`）**完全相同** | 不需要，现有格式直接复用 |
| 原生 App（APK） | `resource-id`（`com.app:id/submit`）或 `accessibility-id` | 需要加 `"platform": "android"` 字段区分选择器类型 |

**重要**：当前阶段做的是"手机浏览器打开网页"，蓝本格式与桌面 Web 测试 **100% 一致**，同一份 `testpilot.json` 两端通用。

#### 日志收集对比

| 类型 | 桌面 Web（Playwright） | 手机 Web（Appium + adb） | 状态 |
|------|----------------------|------------------------|------|
| JS 控制台错误 | `page.on("console")` — 实时精确 | `AndroidController.get_browser_log()` → Appium `POST /log {"type":"browser"}` | ✅ **v5.0-E 已实现** |

## 七、MCP 配置（编程AI双向通信）

### 什么是 MCP
MCP（Model Context Protocol）让编程AI（Windsurf/Cursor）直接调用 TestPilot 的测试能力。
实现闭环：编程AI写代码 → TestPilot测试 → 报Bug → 编程AI修复 → 再测 → 直到0Bug。

### Windsurf 配置
MCP配置文件位置：`~/.codeium/windsurf/mcp_config.json`（即 `C:\Users\用户名\.codeium\windsurf\mcp_config.json`）

在 `mcpServers` 对象里添加以下配置（和其他MCP并列）：

```json
"testpilot": {
  "command": "D:\\projects\\TestPilotAI\\.venv\\Scripts\\python.exe",
  "args": ["D:\\projects\\TestPilotAI\\run_mcp.py"],
  "env": {}
}
```

**⚠️ 注意：必须用 `run_mcp.py` 绝对路径方式，不能用 `-m src.mcp_server`（会有 PYTHONPATH 问题）**

### 新电脑安装 MCP 步骤（完整）
1. 先完成项目初始化（poetry install + playwright install）
2. 用 Windsurf 打开 `~/.codeium/windsurf/mcp_config.json`
3. 在 `mcpServers` 里添加上面的 `testpilot` 配置
4. **修改 command 和 args 里的路径为实际项目路径**
5. 重启 Windsurf（或 `Ctrl+Shift+P` → `Reload Window`）
6. 启动引擎：`poetry run python main.py`
7. 在 Windsurf 聊天里验证：让AI调用 `check_engine_health` 工具

## 七、MCP 踩坑记录（重要！）

开发 MCP Server 过程中遇到的坑，单位电脑安装时注意：

### 坑1：MCP配置文件位置
- ❌ 错误：`~/.windsurf/mcp_config.json`
- ✅ 正确：`~/.codeium/windsurf/mcp_config.json`（即 `C:\Users\用户名\.codeium\windsurf\mcp_config.json`）

### 坑2：模块导入路径
- ❌ 错误配置：`"args": ["-m", "src.mcp_server"]` — MCP进程没有正确的工作目录，会报 ModuleNotFoundError
- ✅ 正确配置：`"args": ["D:\\projects\\TestPilotAI\\run_mcp.py"]` — 用 `run_mcp.py` 入口脚本，里面会自动设置 sys.path

### 坑3：httpx 在 MCP 进程中网络异常
- ❌ 用 `httpx`（同步或异步）都会报 `[WinError 10054] 远程主机强迫关闭连接`
- ✅ 用 Python 标准库 `urllib.request` 替代，零依赖且稳定

### 坑4：PowerShell 的 ConvertTo-Json 破坏格式
- ❌ 用 PowerShell 修改 JSON 会搞乱缩进和字段顺序
- ✅ 用 Python 脚本修改 JSON，或者直接在 Windsurf 里手动编辑

### 坑5：每次修改 MCP Server 代码需要 Reload
- MCP Server 是 Windsurf 启动时加载的独立进程
- 修改 `src/mcp_server.py` 或 `run_mcp.py` 后，必须 `Ctrl+Shift+P` → `Reload Window`
- Reload 会杀掉所有后台进程（引擎也会停），需要重新启动引擎

### 坑6：JSON 文件 BOM 编码
- PowerShell 写入的 JSON 可能带 UTF-8 BOM 头
- Python 读取时需要用 `encoding='utf-8-sig'` 而不是 `'utf-8'`

## 八、蓝本（testpilot.json）格式完整说明

蓝本是编程AI写完功能后生成的测试规范，TestPilot 按蓝本精确执行测试。

### 完整格式

```json
{
  "app_name": "应用名称",
  "base_url": "http://localhost:3000",
  "version": "1.0",
  "pages": [
    {
      "url": "/",
      "title": "页面标题",
      "elements": {
        "输入框": "#todoInput",
        "添加按钮": "#addBtn",
        "列表": ".todo-list"
      },
      "scenarios": [
        {
          "name": "场景名称",
          "description": "场景描述",
          "steps": [
            {
              "action": "navigate",
              "value": "http://localhost:3000",
              "expected": "页面正常加载",
              "description": "打开首页"
            },
            {
              "action": "fill",
              "target": "#todoInput",
              "value": "要输入的文字",
              "description": "输入待办内容"
            },
            {
              "action": "click",
              "target": "#addBtn",
              "description": "点击添加按钮"
            },
            {
              "action": "assert_visible",
              "target": ".todo-item",
              "description": "验证待办项出现"
            },
            {
              "action": "screenshot",
              "expected": "列表中显示刚添加的待办，统计栏数字正确",
              "description": "截图验证状态"
            }
          ]
        }
      ]
    }
  ]
}
```

### 支持的 action 类型

| action | 说明 | 必须字段 |
|--------|------|----------|
| `navigate` | 打开URL | `value`（URL） |
| `click` | 点击元素 | `target`（CSS选择器） |
| `fill` | 输入文字 | `target` + `value` |
| `select` | 下拉选择 | `target` + `value` |
| `wait` | 等待元素出现 | `target` + 可选 `timeout_ms` |
| `screenshot` | 截图+AI验证 | `expected`（预期描述） |
| `assert_visible` | 验证元素可见 | `target` |
| `assert_text` | 验证文本内容 | `target` + `value`（预期文本） |
| `scroll` | 滚动页面 | — |

### v1.3 新增字段
| 字段 | 说明 | 示例 |
|------|------|------|
| `wait_after_ms` | 操作后等待毫秒（处理异步加载/动画） | `"wait_after_ms": 500` |
| `auto:` 前缀 | value字段智能生成测试数据 | `"value": "auto:number:1-999"` |

**auto: 智能输入支持的类型**：
- `auto:number:1-999` — 随机整数
- `auto:number:0.01-99.99` — 随机浮点数
- `auto:text:人名` / `auto:text:公司名` / `auto:text:地址` / `auto:text:句子`
- `auto:email` — 随机邮箱
- `auto:phone` — 随机手机号
- `auto:date` — 随机日期
- `auto:password` — 随机密码（含大小写+数字）

### 关键要点
1. **`target` 必须是精确的CSS选择器**：`#id`、`.class`、`[data-xxx]`、`.parent .child` 等
2. **`expected` 写自然语言**：AI会看截图判断是否符合预期，写越详细越好
3. **一个场景失败不影响后续场景**：TestPilot会继续测完所有场景
4. **Demo蓝本参考**：`demo-app/testpilot.json`（44步，10个场景，覆盖Todo+购物清单）

## 九、重要经验与踩坑

### 环境相关
- **Windows + Uvicorn reload 不兼容 Playwright**：必须 `TP_SERVER_RELOAD=false`
- **pydantic-settings 子配置必须加 env_file**：否则读不到 .env 变量
- **端口统一8900**：前端 vite.config.ts / engineClient.ts / AppLayout.tsx 全部指向 8900

### Playwright 相关
- **Playwright缓存问题**：复用browser context会缓存旧HTML。解决：`Network.clearBrowserCache`（CDP）
- **analyze_screenshot 是同步方法**：不能用 await 调用
- **探索模式 vs 蓝本模式**：探索模式AI猜选择器准确率低（25%）；蓝本模式精确CSS选择器，98%通过率

### Tauri 相关
- **Windows Defender文件锁**：编译时 `.o` 文件被杀毒软件锁定。解决：排除 `src-tauri/target` 目录
- **`CARGO_INCREMENTAL=0`**：禁用增量编译可绕过部分文件锁问题
- **tauri.conf.json**：Tauri 2.x 的 `app` 节点下不能直接放 `title`，只能在 `windows[]` 里

### AI模型相关
- **Doubao-Seed-1.8 价格**：输入0.8/输出2.0元/百万token（含视觉），典型20步测试成本0.017元

**v10.3 小程序自动化连接修复（2026-03-08 下午）**

**问题**：小程序蓝本测试报"桥接服务器启动超时"，连接automator失败。

**根因分析**：
1. 旧版`launch()`在启动时做`cli close`+`cli open`重启，导致模拟器重新初始化（截图显示"模拟器启动失败: TypeError: Failed to fetch"）
2. 桥接服务器启动时在`server.listen`回调里做6次重试连接（阻塞18秒），Python端轮询HTTP时`connected`一直是false
3. Python端只检查`connected=true`，但HTTP服务器虽然在监听却还没连上automator

**关键端口发现（极其重要！）**：
| 端口 | 用途 | 来源 |
|------|------|------|
| **49492** | 开发者工具HTTP服务端口（CLI命令用） | 设置→安全设置→服务端口 |
| **9420** | automator WebSocket端口（自动化用） | `cli auto --auto-port 9420`指定 |
| **9421** | 桥接服务器HTTP端口（Python↔Node通信） | 代码固定 |

**这三个端口是不同的！** `--auto-port 9420`确实有效，不要和服务端口49492混淆。

**修复方案（launch() v8.3 五阶段流程）**：
```
阶段1: cli auto（不重启！避免模拟器崩溃。项目未打开才open+auto）
阶段2: TCP探测WS端口9420就绪（最多20秒）
阶段3: 启动桥接HTTP服务器（不自动连接automator）
阶段4: 等HTTP服务器响应（最多5秒，不管connected状态）
阶段5: POST connect命令主动触发连接（重试8次×5秒=40秒）
```

**修复结果**：
```
cli auto 完成 (rc=0)
WS端口 9420 已就绪（第1秒）
HTTP服务器已就绪（0.5秒）
小程序自动化已连接 | 端口:9420 | 第1次  ← 一次就连上！
```

**修改文件**：
- `src/controller/miniprogram.py`：重写launch()五阶段+_run_cli()+_check_tcp_port()+_kill_bridge()
- `src/controller/miniprogram_bridge_server.js`：v8.3启动时不自动连接，等POST connect命令

**Git提交**：e6aca51

**铁律补充**：
- **不要在launch()里做cli close+cli open重启**：会导致模拟器重新初始化失败
- **桥接服务器启动时不要自动连接automator**：让Python端通过POST主动触发，这样HTTP能立即响应健康检查
- **服务端口≠自动化端口**：设置里的49492是HTTP服务端口，9420是automator端口，别混淆

### v10.4 砍掉桥接服务器+执行器模式（2026-03-08 下午）

**架构变更**：砍掉桥接服务器（`miniprogram_bridge_server.js`），改用一次性Node.js执行器（`miniprogram_runner.js`）。

**新架构**：
```
插件 → Python API → 写JSON步骤文件 → node miniprogram_runner.js → 执行步骤 → 输出JSON → Python解析返回
```
跟`run_blind_test.js`一模一样的模式：Node.js脚本自己做所有事情，跑完就退出。没有长连接、没有桥接服务器、没有端口占用问题。

**三级连接策略**（`miniprogram_runner.js`）：
```
策略1: 直接connect（模拟器已正常+auto已启动时，0.1秒秒连）
策略2: 只执行cli auto → connect（不close不open）
策略3: cli open + cli auto → connect（不做close！）
```

**关键发现和铁律**：

1. **绝对不能cli close**：`cli close`会破坏模拟器状态，之后open回来的模拟器永远是"启动失败: TypeError: Failed to fetch"，连`cli cache --clean compile`都救不回来
2. **开发者工具进程残留问题**：`cli quit`不能杀干净所有进程，必须用`Get-Process wechatdevtools | Stop-Process -Force`彻底杀掉后才能干净重启
3. **模拟器"启动失败"的修复流程**：
   ```
   ① Get-Process wechatdevtools, "微信开发者工具" | Stop-Process -Force
   ② 等5秒
   ③ cli open --project "项目路径"
   ④ 等8秒
   ⑤ cli auto --project "项目路径" --auto-port 9420
   ⑥ 确认输出 √ auto
   ```
4. **run_blind_test.js会断开auto**：该脚本做了`close→open→auto`重连，跑完后`mp.disconnect()`会断开automator连接，后续其他脚本策略1直连会失败
5. **SDK方法全部超时**：`mp.navigateTo()`, `mp.reLaunch()`等SDK封装方法会10秒超时，必须用`evaluate(() => wx.navigateTo({url:...}))`原生API调用
6. **PowerShell Out-File会加BOM头**：导致Node.js的`JSON.parse`失败，需要`replace(/^\uFEFF/, '')`去BOM
7. **项目列表重复**：扫描蓝本时可能存在重复加载，需要去重

- `src/controller/miniprogram_runner.js`（新增）：一次性Node.js执行器，三级连接策略
- `src/api/routes.py`：小程序蓝本测试改用执行器模式，不再依赖MiniProgramController
- `src/controller/miniprogram_bridge_server.js`（保留但不再使用）

### 关键技术发现和铁律

1. **小程序evaluate必须用`new Function`传函数**：
   - `mp.evaluate("字符串")` → 报`Arg string terminates parameters early`或`Unexpected token`
   - `mp.evaluate(new Function("代码"))` → 正确执行，等同于`mp.evaluate(() => { 代码 })`
   - 这是因为automator对字符串参数有严格的解析限制

2. **小程序没有`document`对象**：
   - evaluate里不能用`document.querySelector`，只能用`getApp()`、`getCurrentPages()`、`wx.xxx`
   - 查询DOM元素用automator的`page.$`和`page.$$`（在Node端执行，不在小程序端）
   - 新增`page_query`步骤类型解决此问题

3. **`callMethod('onShow')`会导致100秒超时**：
   - reset_state里调callMethod会让后续截图卡死
   - 解决：去掉callMethod，`reLaunch + sleep(2000)`已足够

4. **策略3必须强杀进程，不能依赖`cli quit`**：
   - 模拟器状态烂了时，`cli quit`也会失败
   - 改为`taskkill /F /IM wechatdevtools.exe /T`强杀所有进程
   - 然后`cli open` → `cli auto`重建干净环境

5. **小程序代码修改后必须重新编译**：
   - 外部编辑器改了js文件，开发者工具不会自动检测
   - 需要在开发者工具里点"编译"，或者执行器自动kill重启

### 10.2 Flutter UI特性（极其重要）

**Flutter渲染出来的Android无障碍树有独特结构：**

1. **所有元素的`text`属性都为空（`text=""`）**，文本内容全部在`content-desc`属性中
2. **按钮是双层结构**：外层`content-desc="btn_login"`(clickable=false) 包裹内层`content-desc="登录"`(clickable=true)
3. **`accessibility_id`策略找到元素后，UiAutomator2会调`waitForIdle()`**，Flutter动画循环会导致该调用永远阻塞

### 10.3 已解决问题（Scene 1 全部通过：7/7步，<8秒）

### 10.4 核心未解决问题

**现象：Scene 2/3 的 `navigate` 后，`findElement` 永远超时（每次请求恰好5秒）**

### 10.16 插件→引擎联调修复：APP不启动 + 停止无效（2026-03-10）

**现象**：
- 插件点击「运行蓝本测试」后，手机上 APP 根本没有打开，引擎疯狂重试 `//android.widget.EditText[@hint='tf_username']` 返回 404
- 点击「停止」按钮，HTTP 返回 200 OK，但测试继续运行不停止
- 用户只能通过停止引擎进程来终止测试

**根因分析（共3个Bug）**：

#### Bug 1：APP 从未启动

**链路追踪**：
1. 插件握手调用 `createMobileSession({ device_name: "V2243A" })` — **只传了 device_name**，没有传 `app_package` / `app_activity`
2. `routes.py create_mobile_session`：`MobileConfig(app_package=req.get("app_package", ""))` → 空字符串
3. `android.py launch()`：`if self._config.app_package and self._config.app_activity:` → **False** → `adb am start` 被跳过
4. APP 从未启动，手机停留在桌面/拨号器，所有元素查找失败

**修复方案**：在 `MobileBlueprintRunner.run()` 执行步骤前，从蓝本读取 `app_package` / `app_activity`，用 `adb shell am start -n {package}/{activity}` 强制启动 APP，等待 3 秒让 Flutter 渲染完成。

#### Bug 2：停止按钮无效

**链路追踪**：
1. 插件发送 `POST /api/v1/test/control?action=stop` → 调用 `test_controller.stop()`
2. `TestController.stop()` 内部检查：`if self._state == TestState.IDLE: return False`
3. `MobileBlueprintRunner` 从未调用 `test_controller.start()` → 状态永远是 IDLE
4. `stop()` 返回 False，`_cancelled` 永远不变，测试无法中断

**修复方案**：`MobileBlueprintRunner.run()` 开始时调用 `test_controller.start(total_steps=...)`，每步调用 `update_progress()` + `wait_if_paused()`，结束时调用 `on_test_complete()`。

#### Bug 3：步骤内重试循环无法中断

**问题**：即使 `_cancelled=True`，`_find_element_with_wait()` 内部有 15 秒重试循环，期间不检查取消标志，必须等循环结束才能响应停止。

**修复方案**：
1. `android.py.__init__` 新增 `self._is_cancelled_fn: object = None`
2. `_find_element_with_wait()` 循环顶部增加检查：`if self._is_cancelled_fn and callable(self._is_cancelled_fn) and self._is_cancelled_fn(): raise RuntimeError("测试已被用户取消")`
3. `MobileBlueprintRunner.run()` 注入回调：`self._ctrl._is_cancelled_fn = lambda: self._test_controller.is_cancelled`

**变更文件清单**：

| 文件 | 变更内容 |
|------|---------|
| `src/testing/mobile_blueprint_runner.py` | +test_controller参数、+APP启动逻辑、+生命周期管理、+取消中断 |
| `src/controller/android.py` | +_is_cancelled_fn属性、+循环内取消检查 |
| `src/api/routes.py` | 两个Android蓝本端点都传入 test_controller |

**铁律**：
- ✅ `MobileBlueprintRunner` 必须调用 `test_controller.start()` — 否则停止/暂停永远不生效
- ✅ 测试前必须用 `adb am start` 确保 APP 在前台 — 不能依赖 `create_mobile_session` 阶段的启动
- ✅ 所有重试循环（`_find_element_with_wait` 等）必须包含取消信号检查
- ⚠️ 插件侧 `createMobileSession` 只传 `device_name` 是已知限制，引擎侧需从蓝本补偿

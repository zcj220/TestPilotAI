## ✅ 插件侧蓝本预检 + 生成提示词强化（2026-03-30）

### 背景

近期蓝本错误高频集中在两类：
1. 编程AI虽然读到了规则文件，但侧边栏复制提示词过短，容易直接凭经验写蓝本，出现裸 `button`、猜测属性值、错误 `flow` 等系统性问题
2. 插件在运行蓝本前没有静态体检，坏蓝本会直接进入引擎，导致用户误以为是引擎稳定性问题

### 修改文件
- `extension/src/sidebarProvider.ts`
- `extension/src/rulesInjector.ts`

### 功能 1 — 运行前蓝本预检

在侧边栏单蓝本/批量蓝本运行前新增静态预检：
- 检测 Web 蓝本中的裸标签选择器（如 `button` / `div` / `span` / `a`）
- 检测 `:contains()`、`nth-child()` 等脆弱或非法选择器
- 对 `[title]`、`[placeholder]`、`[aria-label]`、`[data-testid]`、`[name]` 做源码存在性检查
- 对 `assert_text.expected` 做源码文本存在性检查
- 检测状态切页项目误写 `flow: true` 或伪造 URL 子页面

若发现高风险问题，插件会先弹窗提示，允许用户打开蓝本修正，而不是直接把坏蓝本送进引擎。

### 功能 2 — 复制蓝本生成提示词增强

侧边栏的“复制蓝本生成提示词”从短口令升级为强约束提示，新增硬性要求：
- 先确认真实路由方式，再决定 `flow`
- 禁止裸 `button/a/div/span` 选择器
- 属性选择器值必须能在源码中逐字搜索到
- `assert_text.expected` 必须能对应源码或真实渲染文本
- 状态切页项目禁止编造 `/business`、`/reports` 之类 URL 页面
- 写完后必须做选择器唯一性复查

### 功能 3 — 注入模板版本升级

`rulesInjector.ts` 的模板版本从 v15 升到 v16，新增 `Mandatory Self-Check Before Saving Blueprint` 段落，让注入到用户项目里的 AGENTS.md / copilot-instructions 也同步具备自检约束。

### 设计结论

这次修复的重点不是“帮某一个项目改蓝本”，而是把问题前移到插件层：
- 生成前强化提示
- 运行前静态拦截
- 让错误蓝本更难进入测试链路

## ✅ v14.14 浏览器崩溃自动重建 + Checkpoint 导航路径恢复（2026-03-28）

### 背景

测试过程中发现两个引擎缺陷：
1. 浏览器进程崩溃后引擎仍在运行但页面已失效，用户必须手动重启引擎
2. 步骤失败时 L2 诊断降级到 AI 坐标点击，坐标不准导致继续雪崩；若当前场景有 setup（导航路径），重走 setup 更可靠

### 修改文件
- `src/testing/blueprint_runner.py`：三处修改，语法验证通过

### 功能 1 — 浏览器崩溃自动重建（`_rebuild_browser_if_crashed()`）

**触发条件**：`_execute_step()` 捕获到以下任意关键词异常时，错误前缀标记为 `[BROWSER_CRASH]`：
```
"Target closed" / "Browser was disconnected" / "Connection closed" /
"Target page, context or browser has been closed" / "net::ERR_ABORTED" / "Page closed"
```

**修复流程**：
1. `[BROWSER_CRASH]` 前缀被 `HubAction.RETRY` 分支识别
2. 调用 `_rebuild_browser_if_crashed(blueprint)`：关闭旧 context/browser/playwright（忽略异常）→ 调 `self._browser.launch()` 重建 → 重新挂载控制台/网络监听
3. 若有 setup 步骤：重跑 setup 恢复页面状态，再重试失败步骤
4. 若是 flow 模式无 setup：重跑首场景，再重试

### 功能 2 — Checkpoint 导航路径恢复（RETRY 分支增强）

**触发条件**：步骤失败 + L2 建议 RETRY + 无 `recover_selector`（坐标）+ 场景有 setup 步骤

**修复流程**：
1. 重跑 setup 步骤（确定性 CSS/点击，比 AI 坐标可靠）→ 回到已知正确页面状态
2. 若 setup 重跑失败→ 降级到 L2 提供的坐标点击（`recover_selector`）
3. 最后重试原本失败的步骤

**设计意图**：setup = 页面标签 + 导航路径。凡是通过 2+ 步导航才能到达的场景，必须定义 setup。引擎会自动用 setup 恢复页面状态，让场景具备自愈能力。

### 蓝本写作规则同步更新

`extension/templates/platforms/web.md` (v16→v17 规则补充)：
- EIGHT: Setup — Rule 6：多步导航场景强制定义 setup，引擎自动用其做 checkpoint 恢复

## ✅ v14.12 引擎 + 蓝本选择器修复（2026-03-28）

### 引擎修复：AI L2 max_tokens 500→800

**根因**：L2 诊断 prompt 要求输出 JSON（diagnosis/fault/suggestion/blueprint_fix/resume_step），500 token 经常不够。日志出现 `输出=500 tokens` 截断，导致 JSON 不完整，`json.loads()` 解析失败，L2 返回空字符串，所有恢复机制失效。

**修复**：`ai_hub.py` 中 L2 调用的 `max_tokens` 从 500 提升到 800。

## ✅ v14.11 两处引擎修复（2026-03-28）

### 修复1：AI L2 诊断不知目标（只看过去不看未来）

**根因**：`build_blueprint_steps_context()` 在 `i > current_step_num` 时 `break`，AI L2 收到的蓝本上下文只到当前失败步，完全不知道接下来还要做什么，无法正确判断是应该恢复session还是换策略。

**修复**（`ai_hub.py`）：
- `build_blueprint_steps_context` 额外展示后续3步，标记为 `← 【后续目标】`
- 当前失败步仍标记 `← 【❌当前失败步】`
- prompt 标题从"到当前步为止"改为"当前步 + 后续3步目标"

**效果**：AI L2 能看到失败步后面还要 assert_text / screenshot / close-dialog，从而做出更准确的诊断（如判断是 session丢失 还是选择器错误）。

---

### 修复2：flow模式session丢失后无法恢复（RUN_SETUP 死路）

**根因**：flow模式场景没有 `setup:` 字段，所以 `setup_step_count == 0`。当AI L2诊断出"session丢失"并返回 `run_setup` 时，引擎命中 `else` 分支：`logger.warning("当前场景无setup定义，跳过步骤N")`，所有后续步骤都在登录页面上执行，全部失败。

**修复**（`blueprint_runner.py`）：
- 在原 `else` 分支中，检测 `is_flow and page.scenarios`
- 若满足，使用 `page.scenarios[0].steps`（即当前页第一个场景，通常是登录流程）作为恢复步骤
- 恢复成功后，重试当前失败步骤

```
旧行为：is_flow=True, setup_step_count=0 → logger.warning → 跳过 → 卡死在登录页
新行为：is_flow=True, setup_step_count=0 → 重跑首场景（登录） → 重试当前步骤
```

**注意**：flow模式首场景必须是登录/初始化场景，否则恢复无意义。对于 account_book，首场景是"登录进入公司账本"，符合要求。

---

### 问题说明：Python改动不需要重新构建VSIX

TestPilot插件每次运行测试都会以子进程方式启动 Python 引擎（`python main.py`），保存 `.py` 文件后下一次运行即生效。只有修改 `extension/` 目录下的 TypeScript 文件或 `package.json` 时，才需要重新打包安装 VSIX。

## ✅ v14.10 三处引擎修复（2026-03-28）

### 问题1：AI坐标缓存盲信（导致级联失败）

**根因**：`_try_ai_coord_click()` 缓存命中后直接 `mouse.click(x, y)` 再 `return True`，完全不验证点击是否产生了实际 DOM 变化。若缓存坐标已过期（页面布局改变），则点击了错误位置，返回"成功"，导致后续所有依赖此步骤的操作级联失败。

**修复**：命中缓存后，对比点击前后 `document.body.innerHTML.length`：
- 若长度无变化 → 调用 `web_cache.invalidate_ai_coord()` 清除坏缓存，`return False`，触发新鲜 AI 截图分析
- 若长度有变化 → 验证通过，`return True`

**新增**：`web_cache.py` 添加 `invalidate_ai_coord(page_url, selector)` 方法。

### 问题2：步骤计数分母偏小（`[步骤141/122]`）

**根因**：`blueprint.total_steps` 是蓝本加载时从 JSON 读取的步骤数，不含 `setup` 展开的前置步骤。每个 `enter_business_app` setup ≈ 10 步，8 个场景 × 10 = 80 额外步骤，导致分母 122 远小于实际 141+。

**修复**：`run()` 方法开头预计算 `self._actual_total_steps`（遍历所有页面/场景，加上每个场景的 setup 展开步数）。`[步骤N/M]` 改用 `self._actual_total_steps` 作为分母（在 `_execute_step` 和 flow跳过/取消逻辑中共 3 处）。

### 问题3：引擎版本不可见（无法判断是否在运行旧引擎）

**修复**：测试开始日志新增 `| 引擎=v14.10`，同时显示蓝本步骤数和实际扩展步骤数：
```
蓝本模式测试开始 | 账本 | 页面=1 | 场景=10 | 步骤=122(实际202) | 智能修复=开启 | 引擎=v14.10
```
通过日志中是否出现 `引擎=v14.10` 可立即判断是否运行了最新代码。

## ✅ v14.9 四层降级——智能文本直达（2026-03-28）

### 问题背景

v14.8 修复了 ARIA API 兼容性，但选择器错误仍频繁发生——AI 生成蓝本时很容易写出错误的 CSS 选择器。v14.7/v14.8 的降级链是 CSS → ARIA 快照 → AI 截图，但 ARIA 快照需要 5 秒超时获取整个树，AI 截图需要 3-8 秒 + 1000 token。需要一个**零成本、零延迟**的中间层来处理最常见的选择器错误。

### 核心思路：从 OpenClaw 学到的不只是 ARIA 快照

OpenClaw 不容易出错的根本原因是"**每一步都读页面**"。我们不需要像它那样花 AI 调用，但可以**从已有信息中提取线索**：
- CSS 选择器 `button:has(> svg[class*='Plus'])` 虽然错了，但告诉我们目标是一个 **button**
- description `"点击记一笔按钮"` 告诉我们按钮上有 **"记一笔"** 这个文字
- `input[placeholder='请输入金额']` 直接告诉我们 **placeholder** 是什么

用这些线索直接调用 Playwright 原生定位器（`get_by_text`/`get_by_role`/`get_by_placeholder`），不需要获取 ARIA 快照，零开销！

### 新的四层降级策略

```
第1层 CSS 选择器（0ms, 0 token）→ 95% 成功
  ↓ 失败
第2层 智能文本直达（<100ms, 0 token）→ 挽回 ~80%  ← 新增！
  ↓ 失败
第3层 ARIA Snapshot（200-5000ms, 0 token）→ 挽回 ~60%
  ↓ 失败
第4层 AI 截图坐标（3-8s, 1000+ token）→ 最终兜底
```

### 变更清单

#### 新增 `automator.py` 方法

- `click_by_text(text)` — `page.get_by_text()` 先精确后模糊
- `fill_by_placeholder(placeholder, text)` — `page.get_by_placeholder()` 直达输入框
- `click_by_role_fuzzy(role, name)` — `page.get_by_role()` 模糊匹配
- `fill_by_label(label, text)` — `page.get_by_label()` 通过标签定位

#### 新增 `blueprint_runner.py` 方法

- `_extract_keywords(target, desc)` — 从CSS选择器和description提取定位线索
  - 提取 `:has-text('xxx')` 中的文本
  - 提取 `[placeholder='xxx']` 中的 placeholder
  - 提取 `[aria-label]`/`[title]`/`[name]` 属性值
  - 从元素标签推断 ARIA role（button→button, a→link, input→textbox）
  - 从中文 description 去掉动作词（点击/输入/填写...）提取核心关键词
  - 从引号中提取文字（如"点击'记一笔'按钮"→"记一笔"）
- `_try_smart_text_click(target, desc)` — 三策略顺序尝试：Role+Text → 纯Text → Label
- `_try_smart_text_fill(target, value, desc)` — 三策略顺序尝试：Placeholder → Label → DescLabel
- `_click_with_fallback` / `_fill_with_fallback` 升级为四层

### 以"记一笔"按钮为例的降级模拟

```
CSS: button:has(> svg[class*='Plus'])  → 失败（SVG class 不匹配）
智能文本:
  1. _extract_keywords 提取: role_hint="button", text_hints=["记一笔"]
  2. click_by_role_fuzzy("button", "记一笔") → 成功！page.get_by_role("button", name="记一笔") 精确找到
  ↑ 零ARIA快照、零AI调用、<100ms
```

## ✅ v14.8 ARIA API修复 + 蓝本模板加强（2026-03-28）

### 问题背景

v14.7 实现的三层降级中，ARIA Snapshot 层因使用了已移除的 `page.accessibility.snapshot()` API 而完全失效。实际测试（account_book 项目）发现 ARIA 降级从未成功执行过。同时蓝本模板（web.md）未能阻止 AI 生成错误的 SVG class 选择器和不准确的 assert_text 断言。

### 核心改动

#### 修复 `src/browser/automator.py` — ARIA 快照 API

- **旧代码**：`page.accessibility.snapshot()` → Playwright 1.58 中不存在，抛出 `'Page' object has no attribute 'accessibility'`
- **新代码**：`page.locator("body").aria_snapshot(timeout=5000)` → 返回 ARIA 树文本
- 新增 `_parse_aria_elements()` — 正则解析 ARIA 快照文本，提取 `{role, name}` 列表
- 重写 `_find_aria_node()` — 适配字符串输入，删除旧的 `_collect_aria_candidates()` 递归方法
- `click_by_role()` 和 `fill_by_role()` 保持不变，继续使用 `page.get_by_role()`

#### 加强 `extension/templates/platforms/web.md` — 蓝本生成模板

新增 3 个关键 Pitfall：

| Pitfall | 问题 | 核心规则 |
|---------|------|---------|
| Pitfall 14 | SVG class 选择器永远不对 | 禁止 `svg[class*='xxx']` 和 `button:has(> svg[class*='xxx'])` |
| Pitfall 15 | 图标按钮选择策略 | 有文字的按钮用 `button:has-text('文字')`，提供决策树 |
| Pitfall 16 | assert_text 必须源码对照 | expected 值必须从组件源码中 copy-paste，不得猜测 |

其他模板改动：
- 选择器优先级表：`:has-text()` 从"最后手段"提升为"Tailwind 项目首选"
- Forbidden selectors 列表：新增 `svg[class*='IconName']` 和 `button:has(> svg[class*='xxx'])`
- Gotcha 表（ELEVEN）：新增 SVG class 选择器和 assert_text 猜测两个常见错误

### 为什么 OpenClaw 不容易出这种错？

1. **OpenClaw 每一步都用 AI**：它没有预写选择器，每步都对页面拍 ARIA 快照 → AI 从候选列表中挑元素 → 用 ref 编号精确定位。相当于每一步花 2-5 秒 + 1000 token 换取 100% 准确率
2. **我们用蓝本预写选择器**：零成本零延迟，但选择器的准确率完全依赖 AI 生成蓝本时的质量
3. **v14.8 的解决方案**：通过更严格的模板规则（Pitfall 14-16），在蓝本生成阶段就拦截错误选择器；即使漏网，ARIA 降级层现在真正能工作了

## ✅ v14.7 Web三层降级 + 页面缓存（2026-03-28）

### 问题背景

Web 平台只有 CSS 选择器直连，一旦选择器失效（代码变更/动态DOM），直接走 AI 中枢 L1→L3 决策链。通过分析 OpenClaw (龙虾) 的 Snapshot Ref 系统，发现可以在 CSS 和 AI 中枢之间插入 ARIA Snapshot 降级层，零 AI 成本恢复大多数选择器失效。

### 核心改动：三层降级策略

```
第1层 CSS 选择器（0ms, 0 token）→ 95% 成功
  ↓ 失败
第2层 ARIA Snapshot（200ms, 0 token）→ 挽回 ~90%
  ↓ 失败
```

## ✅ v14.6 L2 run_setup 机制（2026-03-27）

### 问题背景

连续流（flow）场景中，前一个场景完成登录后，下一个场景如果遇到超时/重定向，会被踢回登录页。L2 之前的处理是"猜坐标点登录按钮"——但页面上没有账号密码，点了也无效，照样失败。

### 解决思路

**setup = 可复用的登录模块。** 蓝本里的 `setup` 定义一次登录流程，每个场景引用它；当测试中途 session 丢失时，引擎自动重跑 setup（重新登录），然后继续当前步骤——不需要 AI 猜坐标，不需要改蓝本。

### 实现

#### `src/testing/ai_hub.py`

1. **`HubAction.RUN_SETUP`** 新枚举值：
   ```python
   RUN_SETUP = "run_setup"  # session丢失/跳回登录页：重新执行场景setup步骤后重试
   ```

2. **L2 prompt `flow_analysis_hint`** 新增规则：
   - 检测到当前页面跳转到登录/认证页 → `suggestion` 改为 `run_setup`
   - 含义：引擎静默重跑本场景的 setup 步骤（登录流程）后再重试当前步

3. **L2 JSON `suggestion` 选项** 新增 `run_setup`

4. **L2 结果处理**：`suggestion=="run_setup"` → 返回 `HubDecision(action=HubAction.RUN_SETUP)`

#### `src/testing/blueprint_runner.py`

新增 `elif decision.action == HubAction.RUN_SETUP:` 分支：
- 取 `effective_steps[:setup_step_count]` 静默逐步执行（不计入全局步骤号/all_results）
- 全部成功 → 重试当前失败步骤
- 任一 setup 步骤失败 → 打印警告，跳过当前步
- 当前场景无 setup 定义 → 打印警告，跳过当前步

### 日志示例（修复后）

```

🧠 L2诊断: 当前在登录页，session丢失，建议run_setup

  [setup] navigate → /login
  [setup] fill → input[...] → 13800138000
  [setup] fill → input[...] → test123456
  [setup] click → button[type=submit]
🔑 setup重跑完成，重试步骤58
✓ 步骤58 passed
```

| 需要 L2 知道账号密码 | 是（不可能） | 否（setup 步骤里已有 fill 账号密码） |

## ✅ v14.5 L2诊断精准化——注入蓝本预期流程（2026-03-27）

### 核心改进

**问题背景**：L2 AI（豆包 Seed-1.8）之前只看到"实际执行历史"（过去8步发生了什么），完全不知道蓝本原本计划做什么。这导致诊断结果只能说"登录失败了，先完成登录"，无法指出蓝本哪一步写错了、应该从哪步继续。

**核心思路**：给 L2 AI 同时注入"蓝本预期流程（第1步到当前步）"和"实际执行历史"，让它做对比分析——**找出从哪步开始走偏、为什么走偏、如何修复蓝本 + 如何原地恢复**。

### 变更清单

#### `src/testing/ai_hub.py`

1. **`StepContext` 新增字段**：
   - `blueprint_steps_context: Optional[str]` — 包含蓝本第1步到当前步的格式化摘要，当前失败步标记 `← 【❌当前失败步】`

2. **新增静态方法 `build_blueprint_steps_context(steps, current_step_num)`**：
   - 将蓝本步骤列表格式化为带序号的摘要字符串
   - 最多显示到当前步（不泄露后续未执行步骤）

3. **L2 prompt 结构重构**，新增两个对比分析区块：

   ```
     第2步: click #register-btn  ← 【❌当前失败步】
   
   【实际执行历史（最近几步）】
     第1步: navigate → 成功
     第2步: click #register-btn → 失败(element not found)
   ```
   - 新增 `flow_analysis_hint`：引导 AI 找出从哪步走偏、给恢复坐标 + `resume_step`

4. **L2 JSON 响应新增字段**：

5. **`_blueprint_hints` 收集字段扩展**：
   - 每条 hint 增加 `resume_step` 字段

## 六、测试模式使用策略（重要！务必读懂）

### 6.1 通过率的正确理解

**通过率低 ≠ TestPilot 有问题，而是被测应用有 Bug！**

| 通过率 | 含义 |
|--------|------|
| 83%（30/36步通过） | TestPilot 正常工作，在被测应用中发现了 6 处问题 |
| 100% | 应用功能完全符合蓝本预期，没有 Bug |
| 0%（引擎报错） | 说明引擎/浏览器/环境有问题，需要先排查服务 |

**类比**：体检报告里检出 3 个指标异常，不是体检机器坏了——是身体有问题，体检机器工作越精确越好。

### 6.2 蓝本模式（当前主要模式）

**蓝本模式是日常开发的核心，现阶段几乎只用这个。**

特点：
- 按 `testpilot.json` 里的 CSS 选择器精确操作，不靠 AI 猜页面结构
- 速度快：36 步仅 87 秒
- 结果可重复：每次执行路径完全一致
- 适合：开发中功能验证、提交前回归、CI/CD 自动触发

启动流程：
```powershell
# 1. 启动引擎（新窗口）
Start-Process powershell -ArgumentList '-NoExit','-Command','cd D:\Projects\TestPilotAI; poetry run python main.py'

# 2. 启动浏览器（一次性，重启引擎后需重新执行）
Invoke-RestMethod http://localhost:8900/api/v1/browser/launch -Method POST -Body '{}' -ContentType 'application/json'

# 3. 启动被测应用（例如 BuggyShop）
Start-Process powershell -ArgumentList '-NoExit','-Command','cd D:\Projects\TestPilotAI\test-app; python -m http.server 8080'

# 4. 跑蓝本测试
poetry run python cli.py run -b test-app\testpilot.json -f json -o data\report.json
```

### 6.3 探索模式（暂缓使用）

**探索模式目前不是优先项，有以下限制：**
- AI 自由操作页面，每步都要截图+AI 分析，速度慢（分钟级）
- 结果不确定性高，难以回归对比
- **适合场景**：夜间/闲时让它在后台跑，发现蓝本之外的意外 Bug
- **目前建议**：优先把蓝本写全，覆盖所有核心场景。探索模式留到后续版本或夜间挂机用

### 6.4 真机蓝本测试——屏幕尺寸与选择器

**核心结论：蓝本模式天生支持多机型，不存在屏幕尺寸问题。**

| 测试类型 | 蓝本写法 | 与屏幕尺寸关系 |
|---------|---------|-------------|
| 手机浏览器打开网页 | CSS 选择器（`#addBtn`、`.submit-btn`） | **无关**，浏览器负责布局，选择器永远精确 |
| 原生 App（APK） | resource-id（`com.app:id/btn_submit`）或 accessibility-id | **无关**，元素 ID 不随屏幕变化 |
| 滑动（swipe）操作 | 使用百分比坐标（如"从屏幕 50%位置滑到 20%"）| 需要做适配，用百分比而非固定像素 |

**结论**：同一份蓝本在 5 寸小屏手机和 7 寸平板上跑，结果完全一致，不需要为不同设备写多份蓝本。

**v5.0-E 已完成（2026-03-05）**：新增独立 `POST /api/v1/test/mobile-blueprint` 端点（传 `mobile_session_id`），由 `MobileBlueprintRunner` 通过 Appium + WebView 上下文执行蓝本，同一份 `testpilot.json` 真机可直接运行。

### 6.5 手机蓝本格式 vs 日志收集——详细对比

#### 蓝本格式（testpilot.json）

| 场景 | 选择器格式 | 是否需要新字段 |
|------|-----------|--------------|
| 手机浏览器测 Web 网页 | CSS 选择器（`#addBtn`、`.btn`）**完全相同** | 不需要，现有格式直接复用 |
| 原生 App（APK） | `resource-id`（`com.app:id/submit`）或 `accessibility-id` | 需要加 `"platform": "android"` 字段区分选择器类型 |

**重要**：当前阶段做的是"手机浏览器打开网页"，蓝本格式与桌面 Web 测试 **100% 一致**，同一份 `testpilot.json` 两端通用。

#### 日志收集对比

| 类型 | 桌面 Web（Playwright） | 手机 Web（Appium + adb） | 状态 |
|------|----------------------|------------------------|------|
| JS 控制台错误 | `page.on("console")` — 实时精确 | `AndroidController.get_browser_log()` → Appium `POST /log {"type":"browser"}` | ✅ **v5.0-E 已实现** |
| App 系统日志 | 不适用 | `AndroidController.start_logcat()` → `adb logcat -v threadtime`，守护线程持续收集，步骤失败时自动注入最近 30 行 | ✅ **v5.0-E 已实现** |
| 网络请求错误 | `page.on("response")` — 精确拦截 | adb 抓包或手机浏览器 DevTools（需要额外工具） | ⏳ 暂不支持 |
| 布局溢出检测 | Playwright `page.evaluate()` 运行 JS 检查 DOM | 截图后 AI 判断（没有直接访问 DOM 的手段） | ⚠️ 改为截图分析 |
| App 崩溃/ANR | 不适用 | logcat 自动包含 FATAL/ANR，`get_logcat()` 可过滤 | ✅ **v5.0-E 已实现** |

## 七、MCP 配置（编程AI双向通信）

### 什么是 MCP
MCP（Model Context Protocol）让编程AI（Windsurf/Cursor）直接调用 TestPilot 的测试能力。
实现闭环：编程AI写代码 → TestPilot测试 → 报Bug → 编程AI修复 → 再测 → 直到0Bug。

### Windsurf 配置
MCP配置文件位置：`~/.codeium/windsurf/mcp_config.json`（即 `C:\Users\用户名\.codeium\windsurf\mcp_config.json`）

在 `mcpServers` 对象里添加以下配置（和其他MCP并列）：

```json
"testpilot": {
  "command": "D:\\projects\\TestPilotAI\\.venv\\Scripts\\python.exe",
  "args": ["D:\\projects\\TestPilotAI\\run_mcp.py"],
  "env": {}
}
```

**⚠️ 注意：必须用 `run_mcp.py` 绝对路径方式，不能用 `-m src.mcp_server`（会有 PYTHONPATH 问题）**

### 新电脑安装 MCP 步骤（完整）
1. 先完成项目初始化（poetry install + playwright install）
2. 用 Windsurf 打开 `~/.codeium/windsurf/mcp_config.json`
3. 在 `mcpServers` 里添加上面的 `testpilot` 配置
4. **修改 command 和 args 里的路径为实际项目路径**
5. 重启 Windsurf（或 `Ctrl+Shift+P` → `Reload Window`）
6. 启动引擎：`poetry run python main.py`
7. 在 Windsurf 聊天里验证：让AI调用 `check_engine_health` 工具

## 七、MCP 踩坑记录（重要！）

开发 MCP Server 过程中遇到的坑，单位电脑安装时注意：

### 坑1：MCP配置文件位置
- ❌ 错误：`~/.windsurf/mcp_config.json`
- ✅ 正确：`~/.codeium/windsurf/mcp_config.json`（即 `C:\Users\用户名\.codeium\windsurf\mcp_config.json`）

### 坑2：模块导入路径
- ❌ 错误配置：`"args": ["-m", "src.mcp_server"]` — MCP进程没有正确的工作目录，会报 ModuleNotFoundError
- ✅ 正确配置：`"args": ["D:\\projects\\TestPilotAI\\run_mcp.py"]` — 用 `run_mcp.py` 入口脚本，里面会自动设置 sys.path

### 坑3：httpx 在 MCP 进程中网络异常
- ❌ 用 `httpx`（同步或异步）都会报 `[WinError 10054] 远程主机强迫关闭连接`
- ✅ 用 Python 标准库 `urllib.request` 替代，零依赖且稳定

### 坑4：PowerShell 的 ConvertTo-Json 破坏格式
- ❌ 用 PowerShell 修改 JSON 会搞乱缩进和字段顺序
- ✅ 用 Python 脚本修改 JSON，或者直接在 Windsurf 里手动编辑

### 坑5：每次修改 MCP Server 代码需要 Reload
- MCP Server 是 Windsurf 启动时加载的独立进程
- 修改 `src/mcp_server.py` 或 `run_mcp.py` 后，必须 `Ctrl+Shift+P` → `Reload Window`
- Reload 会杀掉所有后台进程（引擎也会停），需要重新启动引擎

### 坑6：JSON 文件 BOM 编码
- PowerShell 写入的 JSON 可能带 UTF-8 BOM 头
- Python 读取时需要用 `encoding='utf-8-sig'` 而不是 `'utf-8'`

### 支持的 action 类型

| action | 说明 | 必须字段 |
|--------|------|----------|
| `navigate` | 打开URL | `value`（URL） |
| `click` | 点击元素 | `target`（CSS选择器） |
| `fill` | 输入文字 | `target` + `value` |
| `select` | 下拉选择 | `target` + `value` |
| `wait` | 等待元素出现 | `target` + 可选 `timeout_ms` |
| `screenshot` | 截图+AI验证 | `expected`（预期描述） |
| `assert_visible` | 验证元素可见 | `target` |
| `assert_text` | 验证文本内容 | `target` + `value`（预期文本） |
| `scroll` | 滚动页面 | — |

### v1.3 新增字段
| 字段 | 说明 | 示例 |
|------|------|------|
| `wait_after_ms` | 操作后等待毫秒（处理异步加载/动画） | `"wait_after_ms": 500` |
| `auto:` 前缀 | value字段智能生成测试数据 | `"value": "auto:number:1-999"` |

**auto: 智能输入支持的类型**：
- `auto:number:1-999` — 随机整数
- `auto:number:0.01-99.99` — 随机浮点数
- `auto:text:人名` / `auto:text:公司名` / `auto:text:地址` / `auto:text:句子`
- `auto:email` — 随机邮箱
- `auto:phone` — 随机手机号
- `auto:date` — 随机日期
- `auto:password` — 随机密码（含大小写+数字）

### 关键要点
1. **`target` 必须是精确的CSS选择器**：`#id`、`.class`、`[data-xxx]`、`.parent .child` 等
2. **`expected` 写自然语言**：AI会看截图判断是否符合预期，写越详细越好
3. **一个场景失败不影响后续场景**：TestPilot会继续测完所有场景
4. **Demo蓝本参考**：`demo-app/testpilot.json`（44步，10个场景，覆盖Todo+购物清单）

## 九、重要经验与踩坑

### 环境相关
- **Windows + Uvicorn reload 不兼容 Playwright**：必须 `TP_SERVER_RELOAD=false`
- **pydantic-settings 子配置必须加 env_file**：否则读不到 .env 变量
- **端口统一8900**：前端 vite.config.ts / engineClient.ts / AppLayout.tsx 全部指向 8900

### Playwright 相关
- **Playwright缓存问题**：复用browser context会缓存旧HTML。解决：`Network.clearBrowserCache`（CDP）
- **analyze_screenshot 是同步方法**：不能用 await 调用
- **探索模式 vs 蓝本模式**：探索模式AI猜选择器准确率低（25%）；蓝本模式精确CSS选择器，98%通过率

### Tauri 相关
- **Windows Defender文件锁**：编译时 `.o` 文件被杀毒软件锁定。解决：排除 `src-tauri/target` 目录
- **`CARGO_INCREMENTAL=0`**：禁用增量编译可绕过部分文件锁问题
- **tauri.conf.json**：Tauri 2.x 的 `app` 节点下不能直接放 `title`，只能在 `windows[]` 里

### AI模型相关
- **Doubao-Seed-1.8 价格**：输入0.8/输出2.0元/百万token（含视觉），典型20步测试成本0.017元

**v10.3 小程序自动化连接修复（2026-03-08 下午）**

**问题**：小程序蓝本测试报"桥接服务器启动超时"，连接automator失败。

**根因分析**：
1. 旧版`launch()`在启动时做`cli close`+`cli open`重启，导致模拟器重新初始化（截图显示"模拟器启动失败: TypeError: Failed to fetch"）
2. 桥接服务器启动时在`server.listen`回调里做6次重试连接（阻塞18秒），Python端轮询HTTP时`connected`一直是false
3. Python端只检查`connected=true`，但HTTP服务器虽然在监听却还没连上automator

**关键端口发现（极其重要！）**：
| 端口 | 用途 | 来源 |
|------|------|------|
| **49492** | 开发者工具HTTP服务端口（CLI命令用） | 设置→安全设置→服务端口 |
| **9420** | automator WebSocket端口（自动化用） | `cli auto --auto-port 9420`指定 |
| **9421** | 桥接服务器HTTP端口（Python↔Node通信） | 代码固定 |

**这三个端口是不同的！** `--auto-port 9420`确实有效，不要和服务端口49492混淆。

**修复方案（launch() v8.3 五阶段流程）**：
```
阶段1: cli auto（不重启！避免模拟器崩溃。项目未打开才open+auto）
阶段2: TCP探测WS端口9420就绪（最多20秒）
阶段3: 启动桥接HTTP服务器（不自动连接automator）
阶段4: 等HTTP服务器响应（最多5秒，不管connected状态）
阶段5: POST connect命令主动触发连接（重试8次×5秒=40秒）
```

**修复结果**：
```
cli auto 完成 (rc=0)
WS端口 9420 已就绪（第1秒）
HTTP服务器已就绪（0.5秒）
小程序自动化已连接 | 端口:9420 | 第1次  ← 一次就连上！
```

**修改文件**：
- `src/controller/miniprogram.py`：重写launch()五阶段+_run_cli()+_check_tcp_port()+_kill_bridge()
- `src/controller/miniprogram_bridge_server.js`：v8.3启动时不自动连接，等POST connect命令

**Git提交**：e6aca51

**铁律补充**：
- **不要在launch()里做cli close+cli open重启**：会导致模拟器重新初始化失败
- **桥接服务器启动时不要自动连接automator**：让Python端通过POST主动触发，这样HTTP能立即响应健康检查
- **服务端口≠自动化端口**：设置里的49492是HTTP服务端口，9420是automator端口，别混淆

### v10.4 砍掉桥接服务器+执行器模式（2026-03-08 下午）

**架构变更**：砍掉桥接服务器（`miniprogram_bridge_server.js`），改用一次性Node.js执行器（`miniprogram_runner.js`）。

**新架构**：
```
插件 → Python API → 写JSON步骤文件 → node miniprogram_runner.js → 执行步骤 → 输出JSON → Python解析返回
```
跟`run_blind_test.js`一模一样的模式：Node.js脚本自己做所有事情，跑完就退出。没有长连接、没有桥接服务器、没有端口占用问题。

**三级连接策略**（`miniprogram_runner.js`）：
```
策略1: 直接connect（模拟器已正常+auto已启动时，0.1秒秒连）
策略2: 只执行cli auto → connect（不close不open）
策略3: cli open + cli auto → connect（不做close！）
```

**关键发现和铁律**：

1. **绝对不能cli close**：`cli close`会破坏模拟器状态，之后open回来的模拟器永远是"启动失败: TypeError: Failed to fetch"，连`cli cache --clean compile`都救不回来
2. **开发者工具进程残留问题**：`cli quit`不能杀干净所有进程，必须用`Get-Process wechatdevtools | Stop-Process -Force`彻底杀掉后才能干净重启
3. **模拟器"启动失败"的修复流程**：
   ```
   ① Get-Process wechatdevtools, "微信开发者工具" | Stop-Process -Force
   ② 等5秒
   ③ cli open --project "项目路径"
   ④ 等8秒
   ⑤ cli auto --project "项目路径" --auto-port 9420
   ⑥ 确认输出 √ auto
   ```
4. **run_blind_test.js会断开auto**：该脚本做了`close→open→auto`重连，跑完后`mp.disconnect()`会断开automator连接，后续其他脚本策略1直连会失败
5. **SDK方法全部超时**：`mp.navigateTo()`, `mp.reLaunch()`等SDK封装方法会10秒超时，必须用`evaluate(() => wx.navigateTo({url:...}))`原生API调用
6. **PowerShell Out-File会加BOM头**：导致Node.js的`JSON.parse`失败，需要`replace(/^\uFEFF/, '')`去BOM
7. **项目列表重复**：扫描蓝本时可能存在重复加载，需要去重

- `src/controller/miniprogram_runner.js`（新增）：一次性Node.js执行器，三级连接策略
- `src/api/routes.py`：小程序蓝本测试改用执行器模式，不再依赖MiniProgramController

### 关键技术发现和铁律

1. **小程序evaluate必须用`new Function`传函数**：
   - `mp.evaluate("字符串")` → 报`Arg string terminates parameters early`或`Unexpected token`
   - `mp.evaluate(new Function("代码"))` → 正确执行，等同于`mp.evaluate(() => { 代码 })`
   - 这是因为automator对字符串参数有严格的解析限制

2. **小程序没有`document`对象**：
   - evaluate里不能用`document.querySelector`，只能用`getApp()`、`getCurrentPages()`、`wx.xxx`
   - 查询DOM元素用automator的`page.$`和`page.$$`（在Node端执行，不在小程序端）
   - 新增`page_query`步骤类型解决此问题

3. **`callMethod('onShow')`会导致100秒超时**：
   - reset_state里调callMethod会让后续截图卡死
   - 解决：去掉callMethod，`reLaunch + sleep(2000)`已足够

4. **策略3必须强杀进程，不能依赖`cli quit`**：
   - 模拟器状态烂了时，`cli quit`也会失败
   - 改为`taskkill /F /IM wechatdevtools.exe /T`强杀所有进程
   - 然后`cli open` → `cli auto`重建干净环境

5. **小程序代码修改后必须重新编译**：
   - 外部编辑器改了js文件，开发者工具不会自动检测
   - 需要在开发者工具里点"编译"，或者执行器自动kill重启

### 10.2 Flutter UI特性（极其重要）

**Flutter渲染出来的Android无障碍树有独特结构：**

1. **所有元素的`text`属性都为空（`text=""`）**，文本内容全部在`content-desc`属性中
2. **按钮是双层结构**：外层`content-desc="btn_login"`(clickable=false) 包裹内层`content-desc="登录"`(clickable=true)
3. **`accessibility_id`策略找到元素后，UiAutomator2会调`waitForIdle()`**，Flutter动画循环会导致该调用永远阻塞

### 10.3 已解决问题（Scene 1 全部通过：7/7步，<8秒）

| 问题 | 解决方案 |
|------|---------|
| `accessibility_id`策略hang | 转换为`-android uiautomator`策略：`new UiSelector().description("xxx")`绕过`waitForIdle()` |
| Flutter动画阻塞UiAutomator2 | Session创建后设`waitForIdleTimeout=0, waitForSelectorTimeout=0` |
| DialogDismisser与测试竞争 | 测试开始前`self._ctrl._dialog_dismisser.stop()`停止轮询 |
| Flutter键盘遮挡 | `adb shell input keyevent 111`(ESCAPE)收起键盘 |
| `_request()`无超时捕获 | 添加`(TimeoutError, OSError)`异常捕获 |

**现象：Scene 2/3 的 `navigate` 后，`findElement` 永远超时（每次请求恰好5秒）**

- Scene 1 使用初始Session的`launch()` → findElement瞬间返回（0.1秒）
- Scene 2/3 的`navigate`需要重启应用回到登录页 → 重建Session后findElement永远hang 5秒
- 最终结果：**71% (15/21)**，所有元素交互步骤在navigate之后全部失败

**5秒超时来源**：`_find_element()` 的HTTP timeout=5 → `_session_request("POST", "/element", ..., timeout=5)`

**16.6秒总超时算术**：`wait_for_element(timeout_ms=15000)` → 3次重试 × (5s HTTP + 0.5s sleep) = 16.5s ≈ 16.6s

### 10.5 Session重建机制（`_rebuild_session_and_launch()`）

当前`navigate()`中的7步重建流程：
```
1. adb force-stop com.testpilot.flutter_demo（杀应用）
2. DELETE /session/{sid}（3秒超时，失败也继续）
3. am force-stop io.appium.uiautomator2.server + .test（杀UiAutomator2进程）
4. asyncio.sleep(1)（等进程退出）
5. POST /session 创建新Session（与launch()相同capabilities）
6. POST /appium/settings 设 waitForIdleTimeout=0, waitForSelectorTimeout=0
7. adb am start -n com.testpilot.flutter_demo/.MainActivity + asyncio.sleep(3)
```

**Session重建本身是成功的**（可以看到新的Session ID），但新Session创建后的findElement永远hang。

### 10.7 关键矛盾（最重要的线索）

**第一次Session（launch()）→ 一切正常，findElement瞬间返回**

### 10.16 插件→引擎联调修复：APP不启动 + 停止无效（2026-03-10）

**现象**：
- 插件点击「运行蓝本测试」后，手机上 APP 根本没有打开，引擎疯狂重试 `//android.widget.EditText[@hint='tf_username']` 返回 404
- 点击「停止」按钮，HTTP 返回 200 OK，但测试继续运行不停止
- 用户只能通过停止引擎进程来终止测试

**根因分析（共3个Bug）**：

#### Bug 1：APP 从未启动

**链路追踪**：
1. 插件握手调用 `createMobileSession({ device_name: "V2243A" })` — **只传了 device_name**，没有传 `app_package` / `app_activity`
2. `routes.py create_mobile_session`：`MobileConfig(app_package=req.get("app_package", ""))` → 空字符串
3. `android.py launch()`：`if self._config.app_package and self._config.app_activity:` → **False** → `adb am start` 被跳过
4. APP 从未启动，手机停留在桌面/拨号器，所有元素查找失败

**修复方案**：在 `MobileBlueprintRunner.run()` 执行步骤前，从蓝本读取 `app_package` / `app_activity`，用 `adb shell am start -n {package}/{activity}` 强制启动 APP，等待 3 秒让 Flutter 渲染完成。

#### Bug 2：停止按钮无效

**链路追踪**：
1. 插件发送 `POST /api/v1/test/control?action=stop` → 调用 `test_controller.stop()`
2. `TestController.stop()` 内部检查：`if self._state == TestState.IDLE: return False`
3. `MobileBlueprintRunner` 从未调用 `test_controller.start()` → 状态永远是 IDLE
4. `stop()` 返回 False，`_cancelled` 永远不变，测试无法中断

**修复方案**：`MobileBlueprintRunner.run()` 开始时调用 `test_controller.start(total_steps=...)`，每步调用 `update_progress()` + `wait_if_paused()`，结束时调用 `on_test_complete()`。

#### Bug 3：步骤内重试循环无法中断

**问题**：即使 `_cancelled=True`，`_find_element_with_wait()` 内部有 15 秒重试循环，期间不检查取消标志，必须等循环结束才能响应停止。

**修复方案**：
1. `android.py.__init__` 新增 `self._is_cancelled_fn: object = None`
2. `_find_element_with_wait()` 循环顶部增加检查：`if self._is_cancelled_fn and callable(self._is_cancelled_fn) and self._is_cancelled_fn(): raise RuntimeError("测试已被用户取消")`
3. `MobileBlueprintRunner.run()` 注入回调：`self._ctrl._is_cancelled_fn = lambda: self._test_controller.is_cancelled`

**变更文件清单**：

| 文件 | 变更内容 |
|------|---------|
| `src/testing/mobile_blueprint_runner.py` | +test_controller参数、+APP启动逻辑、+生命周期管理、+取消中断 |
| `src/controller/android.py` | +_is_cancelled_fn属性、+循环内取消检查 |
| `src/api/routes.py` | 两个Android蓝本端点都传入 test_controller |

**铁律**：
- ✅ `MobileBlueprintRunner` 必须调用 `test_controller.start()` — 否则停止/暂停永远不生效
- ✅ 测试前必须用 `adb am start` 确保 APP 在前台 — 不能依赖 `create_mobile_session` 阶段的启动
- ✅ 所有重试循环（`_find_element_with_wait` 等）必须包含取消信号检查
- ⚠️ 插件侧 `createMobileSession` 只传 `device_name` 是已知限制，引擎侧需从蓝本补偿

- **现象**：AI返回边缘坐标(x=0.97/y=1.0)时，点击超出窗口范围。Logout按钮y=1.0点到窗口最底部，Add按钮x=0.97超出宽度
- **根因**：`GetWindowRect`+`PrintWindow`截的是完整窗口(416x539含标题栏+边框)，AI坐标基于此计算，但client区域只有400x500。边缘元素坐标本就接近1.0，加上标题栏占比偏移后更易超界
- **试错1**：改`reasoning_effort`从`medium`到`minimal` → 速度快但坐标更不准(y=1.0)
- **试错2**：`clamp(0.03, 0.97)`硬编码限制 → **hack！不同应用/窗口不通用，不专业**
- **正确方案**：截图只截client区域，坐标计算也用client区域。截图和坐标基准完全一致，任何窗口都适用
- **实现**：
  - `window_manager.py` 新增 `get_client_rect()` — `GetClientRect`+`ClientToScreen`获取client屏幕坐标
  - `capture_window()` — 先`PrintWindow`全窗口，再`BitBlt`裁剪client区域
  - `desktop.py` `launch()` — 存储`client_rect`而非`window_rect`
- **坐标链路**：AI看client截图(400x500) → 归一化坐标(0.91) → `client_left + 0.91*400` = 屏幕绝对坐标 → pywinauto: `abs_x - window_left` = 窗口相对坐标（自动包含边框偏移）

#### 坑19：tkinter UI Automation树几乎为空

- **现象**：tkinter的UI Automation树只暴露窗口标题"NoteApp"，内部Entry/Button/Listbox的Name/AutomationId全为空
- **影响**：`get_text()`/`get_visible_text()`只返回"NoteApp"，所有`assert_text`步骤失败
- **方案**：AI视觉OCR fallback——截图发给AI识别所有可见文字
- **文件**：`desktop_blueprint_runner.py` `_get_text()` — 三级降级：UI精确→UI全窗口→AI OCR
- **OCR prompt**：`"请识别这张截图中所有可见的文字内容，逐行列出。只输出文字内容本身。"`
- **效果**：assert_text通过率从0%提升到接近100%

#### 坑20：AI视觉分析极慢（3-4分钟/次）

- **现象**：桌面测试AI调用每次3-4分钟，34步测试需20+分钟。但手动发API只要5-8秒
- **根因**：默认`reasoning_effort="medium"`，视觉分析（图片+复杂提示词）在medium下深度推理
- **方案**：分级设置——坐标定位用`low`(~15s)，OCR用`minimal`(~2s)。不用minimal做坐标（精度不够），不用medium（太慢）
- **文件**：`desktop_blueprint_runner.py` 三处`analyze_screenshot`调用

### 13.9 【重大】AI API超时真正根因（2026-03-14上午）

**现象**：桌面测试AI调用频繁超时（30秒），但Web/Android/小程序测试从不超时，手动调用API 7秒就返回。

**排查过程**：
1. 手动测试API → 7秒返回，排除方舟平台问题
2. 对比不同 `max_completion_tokens` 的耗时：

| max_completion_tokens | 耗时 | 输出tokens | reasoning_tokens | 结果 |
|---|---|---|---|---|
| 65535（全局默认） | 31.9s | 1245 | **1185** | 超时边缘 |
| 4096 | 60s | - | - | 超时 |
| **1024** | **6.0s** | 211 | 152 | ✅ 正常 |
| 256 | 8.4s | 256 | 256 | 响应被截断 |

**根因**：`max_completion_tokens=65535`（全局配置）导致模型**过度思考（reasoning）**！
- 一个简单的坐标定位请求，模型用了1185个reasoning tokens（本该只需要150个）
- reasoning tokens占了93%的输出，实际有效输出只有60 tokens
- Web/Android测试不受影响：它们的prompt场景不同，模型不会过度思考

**修复**：
- `ai_client.py`：`_call_chat` 和 `analyze_screenshot` 新增 `max_tokens` 参数，允许调用方覆盖全局值
- `desktop_blueprint_runner.py`：
  - AI预分析（多元素坐标）：`max_tokens=1024`
  - AI单元素定位：`max_tokens=1024`（512太小，reasoning占满后无实际输出）
  - AI OCR文字识别：`max_tokens=2048`
- `max_retries` 从 `0` 改为 `1`：避免瞬时 `Connection error` 直接失败

**效果**：
- 单次AI调用：32秒 → **6秒**（5倍提升）
- 全量测试：11分钟 → **3分钟**（3.7倍提升）
- 超时率：~30% → **0%**（正常网络下）

**经验教训（铁律）**：
1. **reasoning模型的max_completion_tokens包含reasoning tokens**，设太大会导致模型过度思考
2. **不同场景必须设不同的max_tokens**：简单JSON返回用1024，文字识别用2048，复杂分析才用65535
3. **max_retries=0会让Connection error直接失败**，至少设1次重试
4. **排查API慢时先手动测试排除平台问题**，再对比参数差异

### 13.19 AI定位不稳定根因分析与修复（2026-03-14）

#### 问题现象
桌面蓝本测试25/28通过，3个失败全是AI预分析偶发不返回坐标（如Buy milk），即使蓝本target写法完全正确。

#### 根因分析：各平台AI调用方式对比

| 平台 | AI任务 | reasoning_effort | max_tokens | 为什么不超时 |
|------|--------|-----------------|------------|------------|
| Web | 只做**验证**（截图是否符合预期） | low | 不限制 | 有完整DOM树，CSS选择器直接操作 |
| 安卓 | 只做**验证** | low | 不限制 | 有accessibility树，ID直接操作 |
| 小程序 | 只做**验证** | low | 不限制 | 有WXML树，CSS选择器直接操作 |
| 桌面 | **定位+验证**（看截图找坐标） | low | ~~1024~~ → 不限制 | 无控件树，AI必须看图定位 |

**核心差异**：Web/安卓/小程序的AI**从不需要定位元素**，都通过选择器/ID精确操作。只有桌面AI需要"看图找坐标"，prompt更复杂，reasoning消耗更大。

之前给桌面限制max_tokens=1024是错误的——reasoning吃满token后实际输出为空。改成不限制后跟其他平台一致，AI可以自由思考后输出准确坐标。

#### 修复内容

1. **去掉max_tokens限制**（`desktop_blueprint_runner.py`）— 预分析和单元素定位都不限制，跟Web/安卓一致
2. **坐标容错 `_normalize_coord`**（`desktop_blueprint_runner.py`）— 兼容三种格式：
   - 标准归一化: `0.5` → `0.5`
   - 绝对像素: `219` → `219/dimension`
   - 字符串分数: `"219/889"` → `0.246`
3. **修复venv python启动**（`routes.py`）— `app_exe="python todo_app.py"` 自动检测虚拟环境python，解决系统python缺PyQt5
4. **窗口轮询等待**（`routes.py`）— 从固定3秒改为轮询最多10秒等窗口出现
5. **蓝本规范更新**（`blueprint.md`）— description新增"预期页面变化"要求，编程AI根据源码预测功能行为

#### 修复后测试结果

**26/28通过**，2个失败原因均非AI定位问题：
- Connection error（网络抖动）→ 偶发，非代码问题
- OCR返回不完整（只返回标题"Todo App"）→ OCR质量问题

**AI定位全部正常** — 返回标准归一化坐标如 `{"x": 0.891, "y": 0.131}`，无格式异常。

#### 铁律补充

1. **桌面AI调用不要限制max_tokens** — 跟Web/安卓/小程序一致，让模型自由输出
2. **reasoning_effort="low"是正确选择** — 不思考(minimal)会格式乱，过度思考(medium/high)会慢
3. **坐标容错必须保留** — AI模型升级后输出格式可能变化，容错是防线
4. **app_exe用虚拟环境python** — 系统python通常没有项目依赖
5. **弹窗后必须关闭** — 任何触发弹窗的操作后，蓝本必须加click OK/Yes步骤关闭弹窗，否则遮挡主界面导致后续连锁失败
6. **OCR用low不用minimal** — minimal对复杂界面只读标题，low能识别全部文字

### 13.21 ⭐ 重要下一步：弹窗自愈机制（三层防御）

**问题本质**：蓝本是编程AI写的，即使规范写得再好，AI也可能遗漏弹窗关闭步骤。引擎必须有自救能力，不能被弹窗卡死。

**当前行为**：步骤找不到目标 → 标记Bug → 继续下一步 → 下一步也被弹窗挡 → 连锁失败，整个场景废掉。

**目标行为**：步骤失败 → 检测弹窗 → 自动关闭 → 重试原步骤 → 如果还失败就标记Bug跳到下一场景。

#### 三层防御设计

| 层 | 策略 | 实现方式 |
|---|------|---------|
| **第1层：预防** | 蓝本规范（已做） | workflow要求弹窗后加关闭步骤 |
| **第2层：检测+自愈** | 步骤失败时截图分析 | AI看截图判断"是否有弹窗？OK/Cancel按钮在哪？"，有就自动点击关闭，然后重试原步骤 |
| **第3层：止损** | 场景级跳过 | 自愈失败时，标记当前场景为Bug，跳到下一个场景，不被卡死 |

#### 第2层实现要点
```
步骤执行失败（点击目标找不到/超时）
  → 截图当前界面
  → AI分析："截图中是否有模态弹窗/对话框？如果有，返回关闭按钮(OK/Cancel/X)的坐标"
  → 如果检测到弹窗：
    → 点击关闭按钮
    → 等待500ms
    → 重试原步骤
    → 把"意外弹窗"作为Bug信息记录到报告
  → 如果没有弹窗：
    → 正常走Bug标记流程
```

#### 第3层实现要点
```
当前场景连续失败N步（如3步）
  → 认为场景已不可恢复
  → 标记整个场景为"blocked"
  → 跳到下一个场景的第一步
  → 避免在废掉的场景里浪费AI调用
```

**优先级**：第2层（弹窗自愈）最重要，能解决90%的卡死问题。第3层（场景跳过）是兜底。

### 13.23 AI中枢决策层接入（2026-03-14）

#### 13.23.1 架构设计

创建 `src/testing/ai_hub.py` 作为跨平台统一的AI大脑，所有Runner步骤失败时统一向Hub请求决策。

**分层策略：**
| 层级 | 能力 | 触发条件 | AI成本 |
|------|------|----------|--------|
| L0 | 直通 | 步骤通过 | 0 |
| L1 | 弹窗自愈 | 步骤失败 | ~¥0.01/次 |
| L2 | 失败诊断（预留） | L1未解决 | ~¥0.02/次 |
| L3 | 场景级熔断 | 连续失败≥3 | 0 |

**核心类：**
- `AIHub` - 中枢决策器，持有AI客户端，提供 `on_step_failed()`/`on_step_passed()`/`on_scenario_start()`
- `StepContext` - 步骤上下文数据包，包含 `screenshot_fn`/`click_fn` 平台回调
- `HubDecision` - 决策结果（RETRY/SKIP_STEP/SKIP_SCENE/NONE）
- `DIALOG_BUTTON_KEYWORDS` - 跨平台弹窗按钮关键词集合

#### 13.23.2 三个Runner统一接入

**改造方式：** 每个Runner在`__init__`中创建`self._hub = AIHub(ai_client)`，在步骤循环中：
- 步骤通过 → `hub.on_step_passed()`
- 步骤失败 → 构造`StepContext`（注入平台特定的截图/点击回调）→ `hub.on_step_failed(ctx)` → 根据返回的`HubAction`执行重试/跳过

**平台桥接方法：**
- Desktop: `_hub_screenshot()` → `ctrl.screenshot()`, `_hub_click(nx,ny)` → 窗口坐标转换 → `ctrl._click_at()`
- Web: `_hub_screenshot()` → `browser.screenshot()`, `_hub_click(nx,ny)` → viewport乘法 → `page.mouse.click()`
- Mobile: `_hub_screenshot()` → `ctrl.screenshot()`, `_hub_click(nx,ny)` → 屏幕尺寸乘法 → `ctrl.tap_xy()`

**关键变化：**
- Desktop Runner的`_try_heal_popup()`方法已删除，逻辑迁移到AIHub._try_l1_popup_heal()
- 三个Runner的`_DIALOG_BUTTONS`统一引用`DIALOG_BUTTON_KEYWORDS`
- 三个Runner的`is_dialog_btn`判断统一使用`AIHub.is_dialog_button()`
- 连续失败计数/熔断逻辑全部由Hub管理

#### 13.23.5 归因预判：谁的锅？（v12.2, 2026-03-14）

**问题：** 所有失败都调AI分析 → 浪费token。但"文本不匹配"这种100%是应用Bug，不需要AI看。

**L0.5规则预判（零token）：**
```
步骤失败 → AIHub.on_step_failed()
  → L3 熔断（无AI）
  → L0.5 规则预判（无AI）
       ├─ "文本断言失败: 预期'X', 实际'Y'" → APP Bug，不调AI ✅
       ├─ "计算验证失败" → APP Bug，不调AI ✅
       └─ 其他 → 继续L1/L2（调AI）
  → L1 弹窗自愈（AI截图）
  → L2 失败诊断+归因（AI截图+历史）
```

**新增 `FaultType` 枚举：**
- `APP` — 应用的Bug（数据错误/崩溃/功能异常）→ 报给编程AI
- `TEST` — 测试脚本/框架问题（选择器错/时序/弹窗）→ 中枢自己处理
- `UNKNOWN` — 无法判断

**Bug标签系统：** Runner根据 `decision.fault` 给Bug.category加前缀：
- `[应用Bug]文本断言失败` → 编程AI该修
- `[测试问题]步骤执行错误` → 我们的问题，编程AI不用管

**L2提示词升级：** 要求AI同时判断fault归因 + 中文引号容错修复

**统计新增：** `hub.stats["rule_judgments"]` — 规则预判次数（省了多少次AI调用）

### 16.3 扩展按钮状态 Bug（坑23）

**现象**：点击"一键连接"后，按钮应从绿色变为红色"断开"，但一直保持灰色，最终超时恢复绿色

**根因**：原代码用 `delays = [8000, 16000, 25000]` 三个固定时间点检查，用户**多次点击**时，每次点击都生成一套独立的 `setTimeout` 链，旧的 27s 超时 timer 最后反而把已经变红的按钮**改回了绿色**。另外 8s 才第一次检查，引擎 3-4s 就起来了，等待太久。

**修复**：
- 加 `launchTimeoutId` 变量，每次点击先 `clearTimeout` 取消旧 timer
- 改为每 **3s 检查一次**，最多 8 次（24s），引擎通常 3-4s 内响应

**文件**：`extension/src/sidebarProvider.ts`

### 16.4 经验库"无法连接到引擎"（坑24）

#### 小程序测试步骤进度实时推送
**问题**：小程序测试运行时看不到步骤进度（第1步/第2步...），只有测试完成后才一次性显示结果，导致用户误以为卡住。

**根因**：小程序测试用`subprocess.run`一次性调Node.js执行完再返回，期间没有WebSocket进度推送。Web/桌面/手机测试都有`_on_step`回调实时推送，但小程序没有。

**修复**：
1. `miniprogram_runner.js`已有`[PROGRESS]`和`[STEP]`输出到stderr（每步开始/完成）
2. `routes.py`改`subprocess.run` → `Popen`，线程实时读stderr，协程推送WebSocket：
   - `[PROGRESS] 3/28 点击购物车按钮` → `ws_manager.send_step_start(0, "🔄 3/28 ...")`
   - `[STEP] ✅ #3 passed ...` → `ws_manager.send_log("✅ #3 ...")`
3. 测试开始/完成时也发`send_log`和`send_test_done`

现在小程序测试也能实时看到每步执行状态，跟Web/桌面/手机测试体验一致。

**提交**：`078a29c` fix: 恢复双按钮 + 小程序测试实时步骤进度推送

#### stdout/stderr管道死锁修复
**问题**：小程序测试70步全部跑完后，插件报`fetch failed`，不显示bug列表和步骤详情。

**根因**：改成Popen后，stderr线程和stdout读取是**串行**的——先等stderr读完再读stdout。但Node进程同时往stdout写JSON结果，如果stdout管道缓冲区满（通常64KB），Node进程会阻塞等待stdout被读取，而stderr也不会关闭。形成死锁：
- Python等stderr结束 → stderr等Node退出 → Node等stdout被读取 → Python没在读stdout

**修复**：两个线程同时读stdout和stderr，避免死锁。同时给插件端HTTP POST请求加10分钟超时（`AbortSignal.timeout(600_000)`），防止长时间测试被默认超时中断。

**提交**：`bc80ce4` fix: 修复小程序测试stdout/stderr死锁 + HTTP请求加10分钟超时

#### 小程序步骤偶发卡住+自动重试
**问题**：第2步偶尔卡住不动，但重跑就正常。典型的automator连接时序问题——`reLaunch`后页面刷新，automator的page引用短暂失效，`currentPage()`或`page.$()`挂起。

**修复**（`miniprogram_runner.js`）：每步加15秒整体超时 + 自动重试1次：
- 正常执行 → 15秒超时保护
- 超时 → 等1秒让连接恢复 → 重试（20秒超时）
- 重试也失败 → 标记failed，继续执行后续步骤

**提交**：`92d1727` fix: 小程序每步加15秒超时+自动重试

### 29.4 Token 流程说明

```
登录 → 返回 access_token(1h) + refresh_token(7d)
             ↓ 两个都存入 SecretStorage
             
插件启动检查 →  GET /auth/me（用 access_token）
              ├─ 200 OK → 正常显示已登录状态
              └─ 401 → 自动用 refresh_token 换新 access_token
                       ├─ 成功 → 更新 SecretStorage，继续使用
                       └─ 失败（7天已过）→ 清除所有 token，提示重新登录
```

```env
REQUIRE_EMAIL_VERIFICATION=true

SMTP_PORT=587
SMTP_USER=noreply@example.com
SMTP_PASS=your-smtp-password
SMTP_FROM=TestPilot AI <noreply@example.com>
```

### 29.6 安全注意事项

- Refresh Token 存的是 SHA256 哈希，原始 token 只在登录响应中出现一次，DB 里永远只存哈希
- 每次刷新时旧 Refresh Token 立即作废（轮换机制），防止 token 泄露后被滥用
- 登录失败记录存 DB，服务器重启不影响锁定状态
